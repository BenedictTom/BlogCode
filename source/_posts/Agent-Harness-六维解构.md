## 前言

过去一年，AI Agent 领域最大的认知转变或许不是"模型又变强了"，而是一个更务实的发现：**模型不是瓶颈，Harness 才是**。

有一组来自 Harness Engineering 实践社区的对比数据很能说明问题——在复杂编码任务的基准测试中，将底层模型从 GPT-4o 升级到 Claude 3.5 Sonnet，任务成功率提升了约 8%；但仅仅优化上下文组装策略（不换模型），成功率就提升了 23%；再叠加一层验证循环，又多了 17%。换句话说，围绕模型搭建的"缰绳"（Harness）对最终效果的影响，远大于模型本身的能力跃迁。

这让我们不得不重新审视 Agent 的工程结构。本文试图从一个六维框架出发，拆解 Agent Harness 的完整面貌：

> **Agent Harness = Prompt + Context + Skills + Orchestration + MCP/Tools + Data & Memory**

这六个维度不是互相独立的"模块清单"，而是一个有机系统中的不同截面。Prompt 定义目标，Context 提供视野，Skills 封装能力，Orchestration 编排流程，Tools 执行动作，Memory 沉淀经验。理解它们之间的张力和协作，才是理解 Harness Engineering 的关键。

---

## 一、从 Prompt Engineering 到 Harness Engineering

在展开六个维度之前，有必要先理清一条演化脉络。

**Prompt Engineering** 是最早被广泛讨论的范式。它的核心假设是：只要把指令写得足够精准，模型就能给出好结果。这在单轮对话场景下基本成立，但一旦进入多轮交互、工具调用、长期任务，静态 prompt 就显得力不从心了——你不可能在一段固定文本里预见所有可能的运行时状态。

**Context Engineering** 是下一个阶段。它意识到输入给模型的不只是"指令"，而是整个上下文窗口里的所有信息：系统提示、对话历史、工具返回值、文件内容、环境状态……Context Engineering 的核心工作是**在正确的时刻，把正确的信息，以正确的格式，放进有限的上下文窗口**。这已经是一个工程问题，而非写作问题了。

**Harness Engineering** 则更进一步。它不仅关注"喂给模型什么"，还关注模型输出之后发生的一切：结果是否正确？副作用是否可控？成本是否在预算内？经验是否被留存？Harness 是套在 Agent 外面的整个约束-验证-观测体系。借用控制论的概念，模型是被控对象，Harness 是控制器。

这三个阶段不是替代关系，而是包含关系：好的 Harness Engineering 一定包含精良的 Context Engineering，而 Context Engineering 的基础仍然是写好 Prompt。

## 二、Prompt：从静态指令到动态目标层

在传统认知里，Prompt 是一段写死的系统提示。但在成熟的 Agent 系统中，Prompt 承担的角色更接近控制论中的"目标层"——它定义的不是具体操作步骤，而是**期望的行为边界和价值取向**。

以 Claude Code 为例，它的系统 prompt 并不是一段简单的"你是一个编程助手"。它是一份高度结构化的文档，包含角色定义、行为约束、安全规则、输出格式要求等多个层次。更关键的是，这份 prompt 不是一成不变的——它会根据运行时状态动态组装。比如当用户处于 git 仓库中时，会追加版本控制相关的指令；当检测到特定的项目结构时，会注入对应的编码规范。

这背后的思路是：**Prompt 应该是分层的、可组合的，而不是一整块铁板**。好的 Prompt 设计至少要考虑三个层次：

第一层是**身份与边界**——你是谁，你不能做什么。这是防御性的，确保 Agent 不会越界。

第二层是**行为模式**——面对不同类型的任务，你的默认策略是什么。这是启发式的，为模型提供决策框架。

第三层是**动态注入**——根据运行时上下文（当前文件、项目类型、用户历史偏好）临时补充的信息。这是自适应的，让 Prompt 具备"感知环境"的能力。

从方差管理的角度看，Prompt 层要解决的核心问题是**输入方差**：同一类任务，不同的用户描述方式可能天差地别。好的 Prompt 设计通过明确的约束和格式要求，将这种方差收窄到模型能稳定处理的范围内。

## 三、Context：Agent 的带宽与视野

如果说 Prompt 是 Agent 的"宪法"，Context 就是它的"工作记忆"。上下文窗口是 Agent 在任何一个时刻所能"看见"的全部信息，它的质量直接决定了推理的质量。

### 3.1 上下文的稀缺性问题

即使是 200K token 的窗口，面对一个复杂的代码仓库也远远不够。一个中等规模的项目动辄几十万行代码，加上对话历史、工具返回值，上下文空间捉襟见肘。这就引出了 Context Engineering 的核心挑战：**如何在有限的带宽里传递最大的信息密度**。

Claude Code 对这个问题的工程解法堪称精致。基于源码分析，可以归纳出一条六层渐进式压缩管线（以下为笔者基于源码的归纳性命名，并非源码中的字面标识）：

1. **Tool Result Budget** —— 对工具返回值设置 token 预算，超出部分先截断。这是最粗暴但最高效的第一道防线。
2. **Snip Compact** —— 对超长的单条消息做智能裁剪，保留首尾、省略中间重复部分。
3. **Microcompact** —— 去除消息中的冗余空白和格式噪音，在不丢失语义的前提下节省空间。
4. **Context Collapse** —— 当对话轮次过多时，将早期对话压缩为摘要。这一步用到 LLM 自身来做总结。
5. **System Context Append** —— 在上下文末尾追加关键的系统状态信息，确保模型始终能"看到"当前环境。
6. **Autocompact** —— 作为最后的兜底机制，当总 token 数逼近窗口上限时，触发全局压缩。

这六层不是"选一个用"，而是从上到下**依次尝试**，每一层都只在前一层不够用的时候才启动。这种渐进式策略的精妙之处在于：它在"信息保真度"和"空间效率"之间找到了动态平衡。

### 3.2 上下文漂移

还有一个容易被忽视的问题：**Context Drift**（上下文漂移）。随着对话推进，早期的信息可能已经过时或不再相关，但它们仍然占据着宝贵的上下文空间，甚至可能误导模型的判断。

应对上下文漂移，业界演化出了几种策略：按相关性过滤（只保留与当前任务相关的历史）、滚动摘要（定期将历史压缩为摘要并替换原文）、以及状态外部化（将关键状态存储在外部，需要时再拉回上下文）。Claude Code 的多层压缩管线本质上就是在不断对抗上下文漂移——它通过逐层递进的压缩策略，确保最相关的信息始终留在窗口内。

### 3.3 子 Agent 作为上下文防火墙

一个更激进的思路是：**用子 Agent 隔离上下文**。当一个任务需要大量中间推理（比如分析一个大文件），与其把所有中间结果堆在主对话的上下文里，不如派出一个子 Agent 在独立的上下文空间里完成，然后只把最终结论带回来。

这不仅是一种上下文管理手段，更是一种架构设计模式——它让 Agent 系统具备了类似于操作系统"进程隔离"的能力。子 Agent 的上下文崩溃不会影响主 Agent，主 Agent 的长期记忆也不会被子任务的噪音污染。

## 四、Skills：可复用的能力封装

如果 Prompt 定义了 Agent 的"价值观"，Context 提供了"工作记忆"，那么 Skills 就是 Agent 的"技能证书"——标准化的、可复用的能力包。

### 4.1 为什么需要 Skills？

一个直觉性的问题：既然模型已经"什么都会"了，为什么还需要 Skills？

答案在于**可靠性**和**一致性**。模型确实具备广泛的通用能力，但在特定领域任务上，它的表现方差很大——同一个代码生成任务，十次执行可能产生七种不同的风格。Skills 的本质是**将最佳实践固化为可复用的模板**，从而收窄执行方差。

这和软件工程中"库"和"框架"的理念如出一辙。你不会每次写 HTTP 请求都从 socket 开始，同样，Agent 也不应该每次处理 YAML 文件都要"重新想一遍"怎么做。

### 4.2 Skills 的标准化

以目前生态中较为成熟的实现为例，一个 Skill 通常包含：

**SKILL.md** —— 核心定义文件，用 Markdown 编写，包含 YAML 前置元数据和正文。前置元数据声明技能名称、描述、版本、适用场景等信息；正文是详细的执行指南，告诉 Agent 在什么情况下、按什么步骤、用什么工具来完成任务。

```yaml
---
name: react-component
description: Create production-ready React components
version: 1.0.0
triggers:
  - "create a React component"
  - "build a new component"
---
```

这种结构的精妙之处在于**渐进式披露**（Progressive Disclosure）：Agent 首先只看到技能列表和简短描述，判断是否需要某个技能；如果需要，再加载完整内容。这避免了将所有技能的完整文档塞进上下文——那会迅速耗尽有限的窗口空间。

### 4.3 Skills 生态

Skills 的价值不仅在于单个技能的封装，更在于它建立了一种**可流通的能力交换机制**。一个团队沉淀下来的最佳实践，可以被打包成 Skill 分享给其他团队；一个开源社区积累的经验，可以被任何 Agent 直接使用。

目前已有众多平台和工具采纳了类似的 Skills 规范，形成了一个快速扩展的生态。这种生态化的演进说明一个趋势：**Agent 的能力边界不再取决于单一模型的训练数据，而取决于它能接入多少高质量的 Skills**。

## 五、Orchestration：编排与控制

Orchestration 是 Harness 的骨架。如果把其他五个维度比作器官和肌肉，Orchestration 就是骨骼系统——它决定了 Agent 的"运动方式"。

### 5.1 双层循环架构

从 Claude Code 的源码中可以看到一种经典的编排模式：**会话级循环 + 轮次级循环**的双层架构。

**会话级循环**（对应 Claude Code 中的 QueryEngine）负责管理整个会话的生命周期：初始化上下文、处理用户输入、决定何时结束。它是"外循环"。

**轮次级循环**（对应 queryLoop）是每一轮交互的执行引擎：接收消息、调用模型、解析工具调用、执行工具、将结果反馈给模型……这个内循环会一直转，直到模型决定停止调用工具为止。

这种双层设计的好处在于**关注点分离**——外循环关心"大局"（会话状态、上下文管理、成本控制），内循环关心"细节"（单次工具调用的执行和错误处理）。这和操作系统中"调度器"与"进程"的关系非常相似。

### 5.2 验证循环（Verification Loop）

编排中最容易被忽略但最关键的环节是**验证**。在纯粹的"推理-行动"循环中，Agent 执行完一个操作就直接进入下一步。但在生产环境中，每一步操作的结果都需要被检验：文件是否写入成功？代码是否通过编译？API 调用是否返回了预期结果？

一个完善的验证循环通常提供四种处置策略：

**Retry** —— 结果不符合预期时，用调整后的参数重试。适用于非确定性的操作（比如网络请求超时）。

**Rollback** —— 撤销已执行的操作，恢复到之前的状态。适用于有副作用的写操作（比如文件修改）。

**Abort** —— 直接终止当前任务链，避免错误级联。适用于检测到不可恢复的错误时。

**Human Review** —— 暂停执行，将决策权交还给人类。适用于高风险操作或 Agent 不确定的场景。

验证循环的加入将 Agent 的执行模式从"盲目前进"变成了"走一步看一步"。这正是 Harness Engineering 区别于简单 Prompt Engineering 的地方：**它不仅关心 Agent 做了什么，还关心做得对不对**。

### 5.3 两种编排路线

观察当前主流的 Agent 系统，可以归纳出两种截然不同的编排哲学：

**环境约束型**（以 Codex 为代表）—— 通过严格限制执行环境来保证安全。Agent 运行在沙箱中，网络访问和系统调用受到严格限制，操作原语相对有限，边界是硬性的物理隔离。这条路线的好处是安全保证强、推理开销小；缺点是灵活性受限，难以处理需要广泛系统交互的任务。

**流程治理型**（以 Claude Code 为代表）—— 通过在执行流程中嵌入检查点来管控风险。Agent 拥有丰富的工具集和较大的自由度，但每个操作都要经过权限检查、副作用分类、结果验证等层层关卡。这条路线的好处是灵活性强、能处理复杂任务；代价是工程复杂度高、延迟更大。

两者没有优劣之分，适用场景不同。Codex 的路线更适合对安全性要求极高、任务类型相对固定的场景（比如自动代码审查）；Claude Code 的路线更适合需要 Agent 灵活应对、自主决策的场景（比如复杂的代码重构）。

## 六、MCP/Tools：Agent 的手和脚

如果说模型是 Agent 的大脑，那么 Tools 就是它的手和脚——连接思考与行动的桥梁。

### 6.1 工具接口的设计哲学

一个好的工具接口需要回答三个问题：这个工具能做什么？它是否安全？它的副作用是什么？

Claude Code 的 Tool 接口设计提供了一个值得参考的范本。每个工具都被定义为一个泛型结构 `Tool<Input, Output, P>`，除了基本的输入输出类型定义外，还必须声明几个关键属性：

```typescript
// Claude Code Tool 默认值策略（简化示意）
const TOOL_DEFAULTS = {
  isEnabled: () => true,
  isConcurrencySafe: () => false,   // 默认不安全
  isReadOnly: () => false,           // 默认有副作用
  checkPermissions: (input) =>
    Promise.resolve({ behavior: 'allow', updatedInput: input }),
}
```

**isReadOnly** —— 这个工具是否只读？只读工具可以安全地并发执行，而写操作需要串行化或加锁。

**isConcurrencySafe** —— 是否并发安全？这决定了编排层能否同时调用多个实例。

**checkPermissions** —— 权限检查函数，在执行前决定是否允许此次调用。

最值得注意的是它的**默认值策略**：`isReadOnly = false`、`isConcurrencySafe = false`。这是一种"Fail-Closed"设计——如果工具开发者忘了声明属性，系统会默认它是不安全的、有副作用的，从而给予最严格的限制。只有开发者显式声明安全属性后，才会放宽约束。这比"默认允许"的策略安全得多。

### 6.2 副作用分类

不是所有的工具操作都一样危险。一种实用的分类方法是按副作用的可逆性来划分：

**只读操作**（Read-Only）—— 比如读文件、查询数据库。没有副作用，可以随意重试。

**可逆操作**（Reversible）—— 比如修改文件（可以备份恢复）、创建临时资源（可以删除）。有副作用，但可以回滚。

**不可逆操作**（Irreversible）—— 比如发送邮件、调用外部 API 触发真实业务流程。一旦执行，无法撤回。

**级联操作**（Cascading）—— 比如删除一个数据库表、推送到生产分支。不仅自身不可逆，还会触发连锁反应。

编排层应该根据这四个等级给予不同的管控力度：只读操作自动放行，可逆操作记录检查点以便回滚，不可逆操作需要二次确认，级联操作必须人工审批。这种分级机制让 Agent 在"能力"和"安全"之间找到了务实的平衡点。

### 6.3 MCP：标准化的工具协议

MCP（Model Context Protocol）的意义在于它为 Agent 与外部工具的交互提供了一个**标准化协议**。在 MCP 之前，每个 Agent 框架都有自己的工具注册和调用方式，工具提供者需要为每个框架做适配。MCP 则试图定义一个统一的接口规范——类似于 HTTP 之于 Web 应用的角色。

MCP 和 Skills 有本质区别：**Skills 封装的是"知识"（怎么做），Tools/MCP 封装的是"能力"（做什么）**。一个 Skill 可能会告诉 Agent"创建 React 组件时应该遵循这些规范"，而一个 MCP Tool 则提供"读写文件系统"的实际能力。两者互补，共同构成 Agent 的完整能力栈。

## 七、Data & Memory：Agent 的沉淀层

前面五个维度解决的是 Agent "当下"的问题——怎么理解任务、怎么获取信息、怎么执行操作。而 Data & Memory 解决的是"跨时间"的问题——Agent 如何从过去的经验中学习，如何在多次交互之间保持连贯性。

### 7.1 为什么 Memory 不等于 Context

Context 是当前上下文窗口中的信息，它是短暂的、有界的——每次对话开始时清空，容量受窗口大小限制。Memory 则是持久化的、可检索的——它存储在外部系统中，在需要时被"召回"到上下文中。

两者的关系类似于计算机中的 RAM 与硬盘：Context 是 RAM，速度快但容量小；Memory 是硬盘，容量大但需要显式的读写操作。Harness 的一个重要职责就是管理好这两者之间的"换入换出"。

### 7.2 记忆的层次

在实际的 Agent 系统中，Memory 至少有三个层次：

**会话记忆**（Session Memory）—— 当前对话的完整历史。这是最基础的记忆，通常直接存储在上下文中，通过前文讨论的压缩策略来管理。

**项目记忆**（Project Memory）—— 特定于某个项目或工作空间的长期信息。比如代码仓库的架构约定、团队的编码规范、常见问题的解决方案。Claude Code 中的 `CLAUDE.md` 就是一种项目记忆的实现——它是一份分层的指令与偏好文件（存在全局 `~/.claude/CLAUDE.md`、项目根目录、子目录等多个层级），由用户和 Agent 共同维护，每次启动时自动加载到上下文中。

**全局记忆**（Global Memory）—— 跨项目、跨会话的用户偏好和经验。比如用户喜欢的编码风格、常用的工具链、历史上的决策模式。

从控制论的视角来看，Memory 层对应的是系统的"沉淀层"——它是控制循环中的长期反馈通道。每次任务执行的结果（成功或失败）都应该被沉淀下来，转化为未来决策的参考。这让 Agent 不再是一个无状态的"一次性工具"，而逐渐具备了"成长"的能力。

### 7.3 成本信封（Cost Envelope）

Data & Memory 维度中还有一个常被忽视的要素：**成本管理**。每次 API 调用、每次工具执行、每次上下文扩展都有成本。在生产环境中，一个失控的 Agent 可能在几分钟内消耗掉惊人的 token 预算。

Cost Envelope（成本信封）是一种分层预算机制：在任务级别设置总预算上限，在步骤级别设置单步预算上限，两层约束共同保证 Agent 不会"刷爆信用卡"。当预算即将耗尽时，Agent 应该主动降级——比如减少验证轮次、使用更简洁的上下文、或者直接向用户报告预算不足。

这看起来是个运维问题，但实际上是 Harness 设计的核心关注点之一。一个不考虑成本的 Agent 系统，在 demo 里表现再好，也无法投入生产。

---

## 八、六维之间的张力与协作

回到我们的公式：**Agent Harness = Prompt + Context + Skills + Orchestration + MCP/Tools + Data & Memory**

理解这个公式的关键不是理解每个维度，而是理解它们之间的**张力**：

**Context 与 Skills 的张力**：Skills 内容要加载到上下文中才能生效，但上下文空间有限。渐进式披露（先看摘要再加载全文）就是为了缓解这个矛盾。

**Orchestration 与 Tools 的张力**：更强的编排控制意味着更多的检查点和更长的执行路径，但也意味着更高的延迟和成本。Fail-Closed 的工具默认值设计，就是在这两者之间寻找平衡。

**Prompt 与 Memory 的张力**：Prompt 中的固定指令可能与 Memory 中积累的经验产生冲突。比如 Prompt 说"始终使用 TypeScript"，但项目记忆显示用户近期一直在写 Python。好的 Harness 需要定义清晰的优先级规则来处理这种矛盾。

**效率与安全的根本张力**：更多的验证、更严的权限、更保守的默认值——这些都是安全措施，但也是性能成本。Harness Engineering 的核心艺术就在于：**在给定的安全要求下，找到效率的上界；或者在给定的效率要求下，找到安全的上界**。

---

## 九、结语

Harness Engineering 不是一个全新的学科，它更像是把软件工程中早已存在的智慧——分层架构、关注点分离、Fail-Closed 设计、渐进式降级、可观测性——重新适配到了 AI Agent 这个新场景下。

"Humans steer, Agents execute." 这个理念精确地描述了 Harness 的本质定位：**它不是要取代人类的判断，而是要确保人类的意图能被 Agent 忠实、安全、高效地执行**。

随着模型能力的持续提升，Harness 的重心可能会逐渐从"约束"转向"协作"——从防止 Agent 犯错，到帮助 Agent 更好地理解和实现人类的意图。但无论如何，"缰绳"本身不会消失，它只会变得更加精致。

当我们把目光从"模型有多聪明"转向"缰绳有多可靠"时，Agent 才真正从实验室走向生产环境。而 Prompt、Context、Skills、Orchestration、MCP/Tools、Data & Memory 这六个维度，就是我们理解和构建这根缰绳的基本坐标系。
