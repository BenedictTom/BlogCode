### 1. Java 优先队列的实现原理

**答案：**
Java的PriorityQueue基于堆（Heap）数据结构实现，默认是小顶堆。

**核心原理：**
- 使用数组存储完全二叉树
- 通过`compareTo()`或`Comparator`比较元素大小
- 插入时向上调整（上浮）
- 删除时向下调整（下沉）

**关键方法：**
```java
// 插入元素 - 向上调整
private void siftUp(int k, E x) {
    while (k > 0) {
        int parent = (k - 1) >>> 1;
        Object e = queue[parent];
        if (comparator.compare(x, (E) e) >= 0)
            break;
        queue[k] = e;
        k = parent;
    }
    queue[k] = x;
}
```

**时间复杂度：**
- 插入：O(log n)
- 删除：O(log n)
- 查找最小值：O(1)

### 2. Redis分布式锁原理

**答案：**
Redis分布式锁通过SET命令的NX（不存在才设置）和EX（过期时间）参数实现。

**核心实现：**
```bash
SET key value NX EX seconds
```

**关键特性：**
- **原子性**：SET NX EX是原子操作
- **过期时间**：防止死锁
- **唯一性**：只有第一个客户端能获得锁

**完整实现示例：**
```java
public boolean tryLock(String key, String value, long expireTime) {
    String result = jedis.set(key, value, "NX", "EX", expireTime);
    return "OK".equals(result);
}

public boolean releaseLock(String key, String value) {
    String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                   "return redis.call('del', KEYS[1]) else return 0 end";
    return jedis.eval(script, 1, key, value).equals(1L);
}
```

**注意事项：**
- 必须设置过期时间防止死锁
- 释放锁时要验证锁的持有者
- 考虑锁的续期机制

### 3. 如何用数据库实现一个分布式锁

**答案：**
可以通过数据库的唯一索引和事务特性实现分布式锁。

**方案一：基于唯一索引**
```sql
-- 创建锁表
CREATE TABLE distributed_lock (
    lock_key VARCHAR(100) PRIMARY KEY,
    lock_value VARCHAR(100) NOT NULL,
    expire_time TIMESTAMP NOT NULL,
    create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 获取锁
INSERT INTO distributed_lock (lock_key, lock_value, expire_time) 
VALUES ('order_lock', 'client_001', DATE_ADD(NOW(), INTERVAL 30 SECOND))
ON DUPLICATE KEY UPDATE 
    lock_value = CASE 
        WHEN expire_time < NOW() THEN VALUES(lock_value)
        ELSE lock_value 
    END,
    expire_time = CASE 
        WHEN expire_time < NOW() THEN VALUES(expire_time)
        ELSE expire_time 
    END;
```

**方案二：基于行锁**
```sql
-- 使用SELECT FOR UPDATE
BEGIN;
SELECT * FROM distributed_lock WHERE lock_key = 'order_lock' FOR UPDATE;
-- 执行业务逻辑
COMMIT;
```

**优缺点：**
- **优点**：实现简单，利用数据库事务特性
- **缺点**：性能较低，不适合高并发场景

### 4. ThreadLocal 的数据结构，原理？

**答案：**
ThreadLocal通过ThreadLocalMap存储数据，每个线程都有独立的ThreadLocalMap实例。

**核心结构：**
```java
public class ThreadLocal<T> {
    // ThreadLocal的hash值，用于在ThreadLocalMap中定位
    private final int threadLocalHashCode = nextHashCode();
    
    // 获取当前线程的ThreadLocalMap
    public T get() {
        Thread t = Thread.currentThread();
        ThreadLocalMap map = getMap(t);
        if (map != null) {
            ThreadLocalMap.Entry e = map.getEntry(this);
            if (e != null) {
                @SuppressWarnings("unchecked")
                T result = (T)e.value;
                return result;
            }
        }
        return setInitialValue();
    }
}
```

**ThreadLocalMap结构：**
```java
static class ThreadLocalMap {
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;
        Entry(ThreadLocal<?> k, Object v) {
            super(k);  // 弱引用指向ThreadLocal
            value = v;
        }
    }
    
    private Entry[] table;
    private int size;
}
```

**内存泄漏问题：**
- ThreadLocal被弱引用，可能被GC回收
- 但Entry的value是强引用，可能导致内存泄漏
- 使用完ThreadLocal后要调用remove()方法

### 5. ThreadLocal 如何跨线程通信

**答案：**
ThreadLocal本身无法跨线程通信，但可以通过以下方式实现：

**方案一：TransmittableThreadLocal**
```java
public class TransmittableThreadLocal<T> extends InheritableThreadLocal<T> {
    // 在创建子线程时自动传递值
    @Override
    protected T childValue(T parentValue) {
        return parentValue;
    }
}

// 使用示例
TransmittableThreadLocal<String> context = new TransmittableThreadLocal<>();
context.set("main thread value");

// 子线程可以获取到父线程的值
new Thread(() -> {
    System.out.println(context.get()); // 输出: main thread value
}).start();
```

**方案二：手动传递**
```java
public class ThreadLocalContext {
    private static final ThreadLocal<String> context = new ThreadLocal<>();
    
    public static void set(String value) {
        context.set(value);
    }
    
    public static String get() {
        return context.get();
    }
    
    public static void remove() {
        context.remove();
    }
}

// 在子线程中手动设置
ThreadLocalContext.set("child thread value");
```

**方案三：使用InheritableThreadLocal**
```java
public class InheritableThreadLocal<T> extends ThreadLocal<T> {
    // 子线程创建时会自动继承父线程的值
}
```

### 6. MQ 数据是怎么存储的？

**答案：**
不同MQ的存储机制不同，以Kafka为例：

**Kafka存储机制：**
- **分区存储**：每个topic分为多个partition
- **分段存储**：每个partition分为多个segment文件
- **文件结构**：
  - `.log`：消息数据文件
  - `.index`：偏移量索引文件
  - `.timeindex`：时间索引文件

**存储目录结构：**
```
/tmp/kafka-logs/topic-0/
├── 00000000000000000000.log
├── 00000000000000000000.index
├── 00000000000000000000.timeindex
├── 00000000000000000001.log
├── 00000000000000000001.index
└── 00000000000000000001.timeindex
```

**消息存储格式：**
```
Message Length: 4 bytes
CRC: 4 bytes
Magic: 1 byte
Attributes: 1 byte
Timestamp: 8 bytes
Key Length: 4 bytes
Key: N bytes
Value Length: 4 bytes
Value: N bytes
```

**其他MQ存储方式：**
- **RabbitMQ**：基于Erlang的Mnesia数据库
- **RocketMQ**：基于文件系统，支持多种存储引擎
- **Pulsar**：分层存储，热数据在内存，冷数据在BookKeeper

### 8. 线程池参数详解

**答案：**
线程池的核心参数包括：

**核心参数：**
```java
public ThreadPoolExecutor(
    int corePoolSize,        // 核心线程数
    int maximumPoolSize,     // 最大线程数
    long keepAliveTime,      // 线程空闲时间
    TimeUnit unit,           // 时间单位
    BlockingQueue<Runnable> workQueue,  // 工作队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

**参数详解：**

1. **corePoolSize（核心线程数）**
   - 线程池中会维护的最小线程数
   - 即使这些线程空闲，也不会被回收
   - 建议：CPU密集型任务 = CPU核心数 + 1，IO密集型任务 = CPU核心数 * 2

2. **maximumPoolSize（最大线程数）**
   - 线程池中允许的最大线程数
   - 当工作队列满了，且当前线程数小于最大线程数时，会创建新线程

3. **keepAliveTime（线程空闲时间）**
   - 当线程数大于核心线程数时，多余线程的空闲时间
   - 超过这个时间，多余线程会被回收

4. **workQueue（工作队列）**
   - 存储等待执行的任务的阻塞队列
   - 当所有核心线程都在工作时，新任务会放入队列

**线程池工作流程：**
```
1. 当前线程数 < corePoolSize → 创建新线程执行任务
2. 当前线程数 >= corePoolSize → 任务放入队列
3. 队列满了且当前线程数 < maximumPoolSize → 创建新线程执行任务
4. 队列满了且当前线程数 >= maximumPoolSize → 执行拒绝策略
```

**常用线程池：**
```java
// 固定大小线程池
ExecutorService fixedThreadPool = Executors.newFixedThreadPool(10);

// 可缓存线程池
ExecutorService cachedThreadPool = Executors.newCachedThreadPool();

// 单线程池
ExecutorService singleThreadExecutor = Executors.newSingleThreadExecutor();

// 定时任务线程池
ScheduledExecutorService scheduledThreadPool = Executors.newScheduledThreadPool(5);
```

### 9. 线程池队列什么时候往里加元素？

**答案：**
线程池队列在以下情况下会添加元素：

**添加元素的时机：**
1. **当前线程数 >= corePoolSize**：新任务会先尝试放入队列
2. **队列未满**：任务直接放入队列等待执行
3. **所有核心线程都在工作**：新任务必须排队

**队列类型及特点：**

1. **ArrayBlockingQueue（有界队列）**
```java
// 固定大小的数组队列
BlockingQueue<Runnable> queue = new ArrayBlockingQueue<>(1000);
```
- 特点：FIFO，有界，适合内存敏感场景
- 使用场景：任务量可预测，内存有限

2. **LinkedBlockingQueue（无界队列）**
```java
// 基于链表的无界队列
BlockingQueue<Runnable> queue = new LinkedBlockingQueue<>();
```
- 特点：FIFO，无界，内存占用较大
- 使用场景：任务量不可预测，内存充足

3. **SynchronousQueue（同步队列）**
```java
// 不存储元素的阻塞队列
BlockingQueue<Runnable> queue = new SynchronousQueue<>();
```
- 特点：不存储元素，每个插入操作必须等待另一个线程的移除操作
- 使用场景：要求任务立即执行，不允许排队

4. **PriorityBlockingQueue（优先级队列）**
```java
// 基于优先级的无界阻塞队列
BlockingQueue<Runnable> queue = new PriorityBlockingQueue<>();
```
- 特点：基于优先级排序，无界
- 使用场景：任务有优先级要求

**队列选择建议：**
- **CPU密集型**：ArrayBlockingQueue，避免过多线程切换
- **IO密集型**：LinkedBlockingQueue，提高并发度
- **实时性要求高**：SynchronousQueue，任务立即执行
- **任务有优先级**：PriorityBlockingQueue

### 10. Spring Boot 和 Spring Cloud 的区别？

**答案：**
Spring Boot和Spring Cloud是两个不同层次的技术：

**Spring Boot：**
- **定位**：快速构建Spring应用的脚手架
- **功能**：自动配置、内嵌服务器、生产就绪特性
- **范围**：单个应用/服务

**Spring Cloud：**
- **定位**：分布式系统开发工具集
- **功能**：服务发现、配置管理、熔断器、网关等
- **范围**：微服务架构

**详细对比：**

| 特性 | Spring Boot | Spring Cloud |
|------|-------------|--------------|
| 定位 | 应用开发框架 | 微服务治理框架 |
| 依赖 | spring-boot-starter | spring-cloud-starter |
| 配置 | application.yml | bootstrap.yml + application.yml |
| 服务注册 | 无 | Eureka、Consul、Nacos |
| 配置中心 | 无 | Config、Nacos、Apollo |
| 负载均衡 | 无 | Ribbon、LoadBalancer |
| 熔断器 | 无 | Hystrix、Sentinel |
| 网关 | 无 | Gateway、Zuul |

**使用场景：**
```java
// Spring Boot应用
@SpringBootApplication
public class UserServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(UserServiceApplication.class, args);
    }
}

// Spring Cloud应用
@SpringBootApplication
@EnableDiscoveryClient  // 服务发现
@EnableFeignClients     // 服务调用
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

**关系：**
- Spring Cloud基于Spring Boot构建
- Spring Boot提供基础能力，Spring Cloud提供分布式能力
- 可以单独使用Spring Boot，但不能单独使用Spring Cloud

### 11. 类加载机制？

**答案：**
Java类加载机制包括加载、验证、准备、解析、初始化五个阶段：

**类加载过程：**

1. **加载（Loading）**
   - 通过类名获取类的二进制数据
   - 将二进制数据转换为方法区内的运行时数据结构
   - 在内存中生成Class对象

2. **验证（Verification）**
   - 文件格式验证：魔数、版本号等
   - 元数据验证：语义分析
   - 字节码验证：程序逻辑验证
   - 符号引用验证：符号引用转换为直接引用

3. **准备（Preparation）**
   - 为类变量分配内存
   - 设置类变量的初始值（零值）

4. **解析（Resolution）**
   - 将符号引用转换为直接引用
   - 符号引用：以符号形式描述所引用的目标
   - 直接引用：指向目标的指针、相对偏移量等

5. **初始化（Initialization）**
   - 执行类构造器`<clinit>()`方法
   - 为类变量赋初始值
   - 执行静态代码块

**类加载器层次：**
```
Bootstrap ClassLoader (启动类加载器)
    ↓
Extension ClassLoader (扩展类加载器)
    ↓
Application ClassLoader (应用类加载器)
    ↓
Custom ClassLoader (自定义类加载器)
```

**双亲委派模型：**
```java
public Class<?> loadClass(String name) throws ClassNotFoundException {
    // 1. 检查是否已经加载
    Class<?> c = findLoadedClass(name);
    if (c != null) {
        return c;
    }
    
    // 2. 委派给父加载器
    if (parent != null) {
        try {
            c = parent.loadClass(name);
            if (c != null) {
                return c;
            }
        } catch (ClassNotFoundException e) {
            // 父加载器无法加载
        }
    }
    
    // 3. 自己尝试加载
    return findClass(name);
}
```

**破坏双亲委派的情况：**
- **SPI机制**：如JDBC驱动加载
- **热部署**：如OSGi、Tomcat
- **自定义类加载器**：重写loadClass方法

### 12. JVM 垃圾回收机制？

**答案：**
JVM垃圾回收机制包括垃圾识别、垃圾回收算法、垃圾收集器等：

**垃圾识别算法：**

1. **引用计数法**
   - 每个对象维护一个引用计数器
   - 引用为0时立即回收
   - 缺点：无法解决循环引用问题

2. **可达性分析**
   - 从GC Roots开始，标记所有可达对象
   - 未被标记的对象即为垃圾
   - GC Roots包括：栈中局部变量、静态变量、常量、本地方法栈等

**垃圾回收算法：**

1. **标记-清除（Mark-Sweep）**
```java
// 标记阶段：标记所有可达对象
void mark(Object obj) {
    if (obj != null && !obj.isMarked()) {
        obj.setMarked(true);
        for (Object ref : obj.getReferences()) {
            mark(ref);
        }
    }
}

// 清除阶段：回收未标记对象
void sweep() {
    for (Object obj : heap) {
        if (!obj.isMarked()) {
            free(obj);
        } else {
            obj.setMarked(false);
        }
    }
}
```

2. **复制（Copying）**
   - 将内存分为两块，每次只使用一块
   - 垃圾回收时，将存活对象复制到另一块
   - 优点：效率高，无碎片
   - 缺点：内存利用率低

3. **标记-整理（Mark-Compact）**
   - 标记阶段同标记-清除
   - 整理阶段：将存活对象向一端移动
   - 优点：无碎片，内存利用率高
   - 缺点：效率较低

4. **分代收集**
   - 新生代：复制算法
   - 老年代：标记-整理或标记-清除

**垃圾收集器：**

1. **Serial收集器**
   - 单线程收集器
   - 适合客户端应用

2. **Parallel收集器**
   - 多线程收集器
   - 适合服务器应用

3. **CMS收集器**
   - 并发标记清除
   - 低停顿时间

4. **G1收集器**
   - 区域化内存布局
   - 可预测的停顿时间

5. **ZGC收集器**
   - 低延迟垃圾收集器
   - 停顿时间不超过10ms

**GC调优参数：**
```bash
# 堆内存大小
-Xms4g -Xmx4g

# 新生代大小
-Xmn1g

# 垃圾收集器选择
-XX:+UseG1GC

# GC日志
-XX:+PrintGCDetails -XX:+PrintGCTimeStamps
```

### 13. HashMap 原理

**答案：**
HashMap基于数组+链表+红黑树的数据结构实现：

**核心数据结构：**
```java
public class HashMap<K,V> extends AbstractMap<K,V> {
    // 存储键值对的数组
    transient Node<K,V>[] table;
    
    // 键值对数量
    transient int size;
    
    // 扩容阈值
    int threshold;
    
    // 负载因子
    final float loadFactor;
}

// 链表节点
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;
}
```

**工作原理：**

1. **计算hash值**
```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

2. **确定数组索引**
```java
int index = (n - 1) & hash;  // n是数组长度
```

3. **插入元素**
```java
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    
    // 1. 数组为空，初始化
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;
    
    // 2. 计算索引位置
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    else {
        // 3. 处理hash冲突
        Node<K,V> e; K k;
        if (p.hash == hash && ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        else {
            // 4. 链表插入
            for (int binCount = 0; ; ++binCount) {
                if ((e = p.next) == null) {
                    p.next = newNode(hash, key, value, null);
                    if (binCount >= TREEIFY_THRESHOLD - 1)
                        treeifyBin(tab, hash);
                    break;
                }
                if (e.hash == hash && ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        
        if (e != null) {
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            return oldValue;
        }
    }
    
    // 5. 扩容检查
    if (++size > threshold)
        resize();
    return null;
}
```

**扩容机制：**
```java
final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int oldThr = threshold;
    int newCap, newThr = 0;
    
    if (oldCap > 0) {
        if (oldCap >= MAXIMUM_CAPACITY) {
            threshold = Integer.MAX_VALUE;
            return oldTab;
        }
        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                 oldCap >= DEFAULT_INITIAL_CAPACITY)
            newThr = oldThr << 1; // 阈值翻倍
    }
    
    // 创建新数组，重新hash
    Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
    table = newTab;
    
    // 重新分配元素
    if (oldTab != null) {
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;
                if (e.next == null)
                    newTab[e.hash & (newCap - 1)] = e;
                else if (e instanceof TreeNode)
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                else {
                    // 链表重新分配
                    Node<K,V> loHead = null, loTail = null;
                    Node<K,V> hiHead = null, hiTail = null;
                    Node<K,V> next;
                    do {
                        next = e.next;
                        if ((e.hash & oldCap) == 0) {
                            if (loTail == null)
                                loHead = e;
                            else
                                loTail.next = e;
                            loTail = e;
                        }
                        else {
                            if (hiTail == null)
                                hiHead = e;
                            else
                                hiTail.next = e;
                            hiTail = e;
                        }
                    } while ((e = next) != null);
                    if (loTail != null) {
                        loTail.next = null;
                        newTab[j] = loHead;
                    }
                    if (hiTail != null) {
                        hiTail.next = null;
                        newTab[j + oldCap] = hiHead;
                    }
                }
            }
        }
    }
    return newTab;
}
```

**红黑树转换：**
- 当链表长度 >= 8 且数组长度 >= 64 时，链表转换为红黑树
- 当红黑树节点数 <= 6 时，红黑树转换为链表

**性能特点：**
- 平均时间复杂度：O(1)
- 最坏时间复杂度：O(n)（所有元素hash冲突）
- 空间复杂度：O(n)

### 14. ConcurrentHashMap 原理

**答案：**
ConcurrentHashMap是线程安全的HashMap，JDK 1.8基于CAS + synchronized实现：

**核心数据结构：**
```java
public class ConcurrentHashMap<K,V> extends AbstractMap<K,V> {
    // 存储键值对的数组
    transient volatile Node<K,V>[] table;
    
    // 扩容时的临时数组
    private transient volatile Node<K,V>[] nextTable;
    
    // 基础计数器
    private transient volatile long baseCount;
    
    // 扩容控制
    private transient volatile int sizeCtl;
}

// 节点类型
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    volatile V value;
    volatile Node<K,V> next;
}
```

**线程安全机制：**

1. **CAS操作**
```java
// 使用CAS更新值
static final <K,V> boolean casTabAt(Node<K,V>[] tab, int i, Node<K,V> c, Node<K,V> v) {
    return U.compareAndSwapObject(tab, ((long)i << ASHIFT) + ABASE, c, v);
}

// 使用CAS更新sizeCtl
private final void initTable() {
    Node<K,V>[] tab; int sc;
    while ((tab = table) == null || tab.length == 0) {
        if ((sc = sizeCtl) < 0)
            Thread.yield(); // 其他线程正在初始化
        else if (U.compareAndSwapInt(this, SIZECTL, sc, -1)) {
            try {
                if ((tab = table) == null || tab.length == 0) {
                    int n = (sc > 0) ? sc : DEFAULT_CAPACITY;
                    @SuppressWarnings("unchecked")
                    Node<K,V>[] nt = (Node<K,V>[])new Node<?,?>[n];
                    table = tab = nt;
                    sc = n - (n >>> 2);
                }
            } finally {
                sizeCtl = sc;
            }
            break;
        }
    }
}
```

2. **synchronized锁**
```java
// 对数组元素加锁
synchronized (f) {
    if (tabAt(tab, i) == f) {
        if (fh >= 0) {
            // 链表操作
            binCount = 1;
            for (Node<K,V> e = f;; ++binCount) {
                K ek;
                if (e.hash == hash && ((ek = e.key) == key || (ek != null && key.equals(ek)))) {
                    oldVal = e.val;
                    if (!onlyIfAbsent)
                        e.val = value;
                    break;
                }
                Node<K,V> pred = e;
                if ((e = e.next) == null) {
                    pred.next = new Node<K,V>(hash, key, value, null);
                    break;
                }
            }
        }
        else if (f instanceof TreeBin) {
            // 红黑树操作
            Node<K,V> p;
            binCount = 2;
            if ((p = ((TreeBin<K,V>)f).putTreeVal(hash, key, value)) != null) {
                oldVal = p.val;
                if (!onlyIfAbsent)
                    p.val = value;
            }
        }
    }
}
```

**put操作流程：**
```java
public V put(K key, V value) {
    return putVal(key, value, false);
}

final V putVal(K key, V value, boolean onlyIfAbsent) {
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode());
    int binCount = 0;
    
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        
        // 1. 初始化table
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        
        // 2. 计算索引位置
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            // 3. CAS插入新节点
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;
        }
        
        // 4. 处理hash冲突
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            V oldVal = null;
            synchronized (f) {
                if (tabAt(tab, i) == f) {
                    if (fh >= 0) {
                        // 链表操作
                        binCount = 1;
                        for (Node<K,V> e = f;; ++binCount) {
                            K ek;
                            if (e.hash == hash && ((ek = e.key) == key || (ek != null && key.equals(ek)))) {
                                oldVal = e.val;
                                if (!onlyIfAbsent)
                                    e.val = value;
                                break;
                            }
                            Node<K,V> pred = e;
                            if ((e = e.next) == null) {
                                pred.next = new Node<K,V>(hash, key, value, null);
                                break;
                            }
                        }
                    }
                    else if (f instanceof TreeBin) {
                        // 红黑树操作
                        Node<K,V> p;
                        binCount = 2;
                        if ((p = ((TreeBin<K,V>)f).putTreeVal(hash, key, value)) != null) {
                            oldVal = p.val;
                            if (!onlyIfAbsent)
                                p.val = value;
                        }
                    }
                }
            }
            
            if (binCount != 0) {
                if (binCount >= TREEIFY_THRESHOLD)
                    treeifyBin(tab, i);
                if (oldVal != null)
                    return oldVal;
                break;
            }
        }
    }
    
    // 5. 更新计数
    addCount(1L, binCount);
    return null;
}
```

**扩容机制：**
- 支持多线程并发扩容
- 使用ForwardingNode标记正在扩容的节点
- 其他线程可以参与扩容过程

**性能特点：**
- 读操作无锁，性能高
- 写操作使用分段锁，减少锁竞争
- 支持高并发读写

### 16. SQL 优化方式？越详细越好

**答案：**
SQL优化是一个系统工程，需要从多个维度进行优化：

**1. 索引优化**

**创建合适的索引：**
```sql
-- 单列索引
CREATE INDEX idx_user_name ON users(name);

-- 复合索引（注意列顺序）
CREATE INDEX idx_user_status_created ON users(status, created_at);

-- 覆盖索引
CREATE INDEX idx_user_covering ON users(id, name, email, status);

-- 部分索引
CREATE INDEX idx_user_active ON users(email) WHERE status = 'active';

-- 函数索引
CREATE INDEX idx_user_lower_email ON users(LOWER(email));
```

**索引设计原则：**
- 最左前缀原则：复合索引的列顺序很重要
- 选择性原则：选择区分度高的列建立索引
- 覆盖索引：尽量让查询只通过索引就能完成
- 避免冗余索引：删除不必要的索引

**2. 查询语句优化**

**避免SELECT ***
```sql
-- 不推荐
SELECT * FROM users WHERE status = 'active';

-- 推荐
SELECT id, name, email FROM users WHERE status = 'active';
```

**使用EXISTS代替IN**
```sql
-- 不推荐
SELECT * FROM orders WHERE user_id IN (SELECT id FROM users WHERE status = 'active');

-- 推荐
SELECT * FROM orders o WHERE EXISTS (
    SELECT 1 FROM users u WHERE u.id = o.user_id AND u.status = 'active'
);
```

**避免隐式类型转换**
```sql

SELECT * FROM users WHERE phone = 13800138000;

-- 推荐
SELECT * FROM users WHERE phone = '13800138000';
```

**使用LIMIT分页**
```sql
-- 推荐使用游标分页
SELECT * FROM orders 
WHERE id > 1000 
ORDER BY id 
LIMIT 20;

-- 避免OFFSET分页（大数据量时性能差）
SELECT * FROM orders 
ORDER BY id 
LIMIT 20 OFFSET 1000;
```

### 17. MyBatis 的查询原理？

**答案：**
MyBatis 通过将 Mapper 映射为 `MappedStatement`，由执行器 `Executor` 驱动 `StatementHandler / ParameterHandler / ResultSetHandler` 完成 SQL 组装、参数绑定与结果映射，并内置一级/二级缓存与插件拦截链。

**核心组件与职责：**
- `SqlSessionFactory`/`SqlSession`：会话创建与管理
- `MappedStatement`：一次增删改查的元数据（SQL、参数、返回类型、缓存配置）
- `Executor`：执行器，Simple/Reuse/Batch，负责缓存与查询更新
- `StatementHandler`：预编译、设置参数、执行 SQL
- `ParameterHandler`：参数到 JDBC 的映射
- `ResultSetHandler`：将结果集映射为对象
- `TypeHandler`：Java ↔ JDBC 类型转换
- 插件拦截点：`Executor`、`StatementHandler`、`ParameterHandler`、`ResultSetHandler`

**执行流程：**
1) 解析 `mybatis-config.xml` 与 Mapper XML，构建 `Configuration`
2) 将每条 `<select|insert|... id>` 解析为 `MappedStatement`
3) `SqlSession.getMapper()` 通过 JDK 动态代理生成 Mapper 代理
4) 调用方法 → 定位 `MappedStatement` → 构建 `BoundSql`（动态 SQL + OGNL）
5) `Executor` 查一级缓存（`localCache`，作用域为 `SqlSession`）
6) 未命中则 `StatementHandler` 预编译、`ParameterHandler` 绑定参数，执行 SQL
7) `ResultSetHandler` 映射结果，写入一级缓存；若配置二级缓存且命中策略允许，则参与二级缓存

**示例：**
```xml
<!-- UserMapper.xml -->
<mapper namespace="com.demo.UserMapper">
  <select id="selectById" parameterType="long" resultType="com.demo.User">
    SELECT id, name, email FROM t_user WHERE id = #{id}
  </select>

  <select id="listByStatus" resultType="com.demo.User">
    SELECT id, name, email FROM t_user
    <where>
      <if test="status != null">AND status = #{status}</if>
      <if test="name != null">AND name LIKE CONCAT('%', #{name}, '%')</if>
    </where>
    ORDER BY id DESC
    LIMIT #{limit} OFFSET #{offset}
  </select>
  
  <cache eviction="LRU" flushInterval="600000" size="512" readOnly="false"/>
</mapper>
```

```java
// UserMapper.java
public interface UserMapper {
    User selectById(long id);
    List<User> listByStatus(@Param("status") Integer status,
                            @Param("name") String name,
                            @Param("limit") int limit,
                            @Param("offset") int offset);
}
```

### 18. 活跃用户如何定义？

**答案：**
依据业务定义活跃标准，并落到可计算的事件集合上，常见口径：
- DAU/WAU/MAU：当日/周/月内有“有效行为”的去重用户数
- Retention：在基准日活跃，且 N 日后仍活跃的比例
- 有效行为示例：登录、下单、支付、内容发布、停留时长≥X、有效交互≥Y

**DAU 计算示例（MySQL）：**
```sql
-- 以登录/下单/支付等为有效事件
SELECT COUNT(DISTINCT user_id) AS dau
FROM user_events
WHERE event_time >= CURDATE()
  AND event_time <  DATE_ADD(CURDATE(), INTERVAL 1 DAY)
  AND event_type IN ('login','order_paid','publish');
```

**7 日留存（D0→D7）：**
```sql
WITH d0 AS (
  SELECT DISTINCT user_id
  FROM user_events
  WHERE event_time >= '2025-08-01' AND event_time < '2025-08-02'
), d7 AS (
  SELECT DISTINCT user_id
  FROM user_events
  WHERE event_time >= '2025-08-08' AND event_time < '2025-08-09'
)
SELECT (SELECT COUNT(*) FROM d0 JOIN d7 USING(user_id)) / (SELECT COUNT(*) FROM d0) AS retention_7d;
```

**口径落地建议：**
- 明确“有效行为”与去重维度（user_id/设备/账号）
- 统一时间边界（UTC/本地时区、自然日 vs 滚动 24h）
- 用物化表/定时任务沉淀 DAU/WAU/MAU，供实时看板

### 19. 有没有关注缓存命中情况？

**答案：**
需要对本地缓存与分布式缓存分别监测，常见指标：命中率、Miss、加载耗时、淘汰/过期数、Key 数、内存用量。

**Caffeine（本地缓存）：**
```java
Cache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(100_000)
    .recordStats()
    .expireAfterWrite(Duration.ofMinutes(10))
    .build();

CacheStats stats = cache.stats();
double hitRate = stats.hitRate();
long evictions = stats.evictionCount();
```

**Redis（分布式缓存）：**
```bash
# 基础命中指标
redis-cli INFO stats | egrep 'keyspace_hits|keyspace_misses'
# 内存与淘汰
redis-cli INFO memory | egrep 'used_memory_human|maxmemory_policy'
```

**Prometheus 监控建议：**
- 应用侧暴露：`cache_requests_total{result="hit|miss"}`、`cache_latency_seconds`、`cache_evictions_total`
- Redis Exporter：`redis_keyspace_hits_total / (hits + misses)`、`redis_evicted_keys_total`

**优化思路：**
- 提升命中：热点 Key 预热、恰当 TTL、避免随机 Key（可哈希归一化）
- 降低穿透：布隆过滤器/空值缓存
- 限制击穿：互斥锁/请求合并/异步刷新
- 防雪崩：过期时间加随机抖动、限流与降级策略

### 20. Spring 单例管理怎么做的？

**答案：**
Spring 默认 `singleton` 作用域，容器内单例。核心由 `DefaultListableBeanFactory` 管理，三级缓存解决循环依赖：
- `singletonObjects`（一级）：完全初始化的单例
- `earlySingletonObjects`（二级）：提前暴露的单例实例（未增强）
- `singletonFactories`（三级）：ObjectFactory，用于创建早期引用（AOP 代理）

**生命周期（简化）：** 实例化 → 属性填充 → `Aware` 回调 → `BeanPostProcessor` before → 初始化 → `BeanPostProcessor` after → 放入一级缓存

**示例：**
```java
@Component
@Scope(ConfigurableBeanFactory.SCOPE_SINGLETON) // 默认即为 singleton
public class OrderService { }

@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public class PrototypeTask { }
```

**要点：**
- 单例线程安全需业务自行保证（无状态或自行加锁）
- 原型 Bean 注入单例时如需每次新建，可用 `ObjectProvider`/`@Lookup`

### 22. 如何实现高性能的分布式配置中心？

**答案：**
设计目标：高可用、低延迟、强/最终一致可选、灰度可控、可观测。

**核心设计：**
- 推送/长轮询：客户端与服务端保持长轮询或 SSE/WS，配置变更即时下发
- 客户端本地快照：磁盘 Snapshot 作为降级兜底，启动优先读本地
- 版本与灰度：版本号 + 灰度规则（按 IP/实例/标签），可回滚
- 多级缓存：配置中心内存缓存 → 客户端内存缓存
- 多租户隔离：Namespace + Group + DataId 三级维度
- 限流与背压：批量推送、合并通知、客户端指数退避
- 可观测：发布审计、变更追踪、客户端 ACK 监控、延迟分布

**客户端长轮询示意：**
```bash
POST /v1/cs/configs/listener
Headers: Long-Pulling-Timeout: 30000ms
Body: dataIds=xxx&group=DEFAULT_GROUP&tenant=ns-001&md5=当前本地MD5
# 服务器比较 MD5，不一致则返回变更列表；一致则阻塞至超时或有变更
```

**性能优化要点：**
- Server：分区分片、多副本、读写分离、热点 Key 拆分、批量/异步落盘
- Client：合并多 DataId 请求、指数退避重试、监听去抖动/限频
- 存储：SSD、顺序写、压缩与合并、冷热分层
- 网络：HTTP/2、连接池、零拷贝

**落地选型建议：** Nacos/Apollo/Spring Cloud Config 均可；对强一致/审计要求高选 Apollo，对生态一体化与服务发现合并选 Nacos。
