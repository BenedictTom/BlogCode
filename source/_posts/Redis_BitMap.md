## BitMap

### 位
位（bit）是计算机中最小的数据存储单位，它只有两种状态：0 和 1，就像开关的“开”和“关”。

### 通俗易懂的解释

想象一下，你有一个只能装一枚硬币的小盒子：

- **状态 0**：盒子里没有硬币，表示“关”或“假”。
- **状态 1**：盒子里有一枚硬币，表示“开”或“真”。

多个这样的小盒子排成一排，就可以表示更复杂的信息。例如，用 8 个小盒子（8 位）可以组合出 256 种不同的状态，从 `00000000` 到 `11111111`。

### 底层原理

#### 电子元件实现

在计算机内部，位的状态由电子元件（如晶体管）控制：

- **状态 0**：晶体管断开，电路中没有电流，代表“关”。
- **状态 1**：晶体管导通，电路中有电流，代表“开”。

#### 二进制表示

计算机使用二进制数系统，所有信息（数字、文字、图像等）都被转换为 0 和 1 的序列。例如，数字 5 的二进制表示为 `00000101`。

#### 存储和运算

- **存储**：多个位组成一个存储单元，如字节（8 位）。计算机内存由大量这样的存储单元组成，用于保存数据。
- **运算**：计算机通过逻辑门电路（如与门、或门、非门）对位进行运算，实现加法、减法、逻辑判断等操作。

### 示例

假设我们要存储字母 "A"，其 ASCII 码为 65，二进制表示为 `01000001`：

- 在内存中，需要 8 个位来存储这个信息。
- 每个位的状态对应二进制中的 0 或 1，共同构成了字母 "A" 的编码。


### 通俗解释
想象一下你有一大堆的小盒子，每个盒子只能放一颗弹珠。这些小盒子整齐地排列在一起，形成了一大片的存储区域。BitMap（位图）就像是这样一个由很多个小盒子组成的区域，不过这里的“小盒子”实际上是计算机内存中的一个个位（bit），每个位只有两种状态：有弹珠（1）或没有弹珠（0）。这里，“弹珠存在”表示某个特定的信息为真或者存在；“没有弹珠”则代表信息为假或者不存在。

### 用途
1. **快速查找**：比如你想知道某项信息是否存在，只需看对应位置的位是1还是0就能立刻得到答案。
2. **空间效率高**：由于它使用位作为基本单位，所以能非常高效地利用存储空间来记录大量的布尔值（true/false信息）。

### 简单例子
假设我们有一个BitMap用来记录一周中哪天有会议。如果用最左边的位代表周一，最右边的位代表周日，那么：

- `1010101` 表示周一、周三、周五和周日有会议；
- `0000000` 表示这一周都没有会议安排；
- `1111111` 则意味着每天都有会议。



### ✅ Redis Bitmap 常用命令（基于字符串的位操作）

Redis 的 Bitmap 实际上是基于字符串（String）数据类型实现的，支持对字符串中的每一位进行操作。

#### 1. `SETBIT key offset value`
- **作用**：设置某个 key 的指定位（offset）为 1 或 0。
- **说明**：
  - `offset` 是位的索引，从 0 开始。
  - `value` 只能是 0 或 1。
- **示例**：
  ```bash
  SETBIT user:login:20240801 100 1
  ```
  表示：在 key 为 `user:login:20240801` 的位图中，将第 100 位设置为 1（比如表示用户ID为100的用户今天登录了）。

#### 2. `GETBIT key offset`
- **作用**：获取某个 key 的指定位的值（0 或 1）。
- **示例**：
  ```bash
  GETBIT user:login:20240801 100
  ```
  返回 1 表示该用户登录了，返回 0 表示没登录。

#### 3. `BITCOUNT key [start end]`
- **作用**：统计 key 中值为 1 的位的个数（即“有多少个用户登录了”）。
- **示例**：
  ```bash
  BITCOUNT user:login:20240801
  ```
  返回结果是今天登录的用户总数。

#### 4. `BITOP operation destkey key [key ...]`
- **作用**：对一个或多个位图进行位运算（AND、OR、NOT、XOR），结果存入 destkey。
- **常用操作**：
  - `BITOP AND dest user:login:20240801 user:login:20240802`：求两天都登录的用户。
  - `BITOP OR dest user:login:20240801 user:login:20240802`：求两天中任意一天登录的用户。

#### 5. `BITPOS key value [start] [end]`
- **作用**：查找第一个值为 0 或 1 的位的索引。
- **示例**：
  ```bash
  BITPOS user:login:20240801 1 0 -1
  ```
  查找第一个登录用户的 ID（即第一个值为 1 的位）。

---

### 🌰 实际应用场景举例：用户签到系统

```bash
# 用户ID为10001的用户今天签到（设置第10001位为1）
SETBIT user:sign:20240801 10001 1

# 查询用户是否签到
GETBIT user:sign:20240801 10001

# 统计今天有多少人签到
BITCOUNT user:sign:20240801
```

---

### 🔔 注意事项
- Redis 的 Bitmap 最大支持 2³² 位（约 512MB），可以表示超过 40 亿个用户状态。
- 位图非常适合用于**海量数据的布尔状态记录**，比如签到、活跃用户统计、去重等。


非常好的问题！你抓住了 Redis Bitmap 的一个**核心机制**：**它是如何处理“位偏移（offset）超出当前字符串长度”的情况的？**

我们来一步步解释，用你举的例子：“如果 key 的字符串没有那么多位，比如 key 就是一个 '1'”。

---

### 🌰 举个例子：从头开始

假设你现在执行：

```bash
SET mykey 1
```

这时候，`mykey` 对应的字符串值是 `"1"`，它在内存中是一个字节（ASCII 码 49），二进制是：

```
00110001
```

（这是字符 `'1'` 的 ASCII 二进制表示）

---

### ✅ 现在你执行一个位操作，比如：

```bash
SETBIT mykey 10 1
```

意思是：把 `mykey` 的第 10 位（从 0 开始）设置为 1。

但当前字符串只有 8 位（1 字节），第 10 位还不存在！

---

### 🎯 Redis 的处理方式：**自动扩展 + 零填充**

Redis 会：

1. **自动扩展**字符串的长度，直到能容纳第 10 位。
2. **中间空缺的位用 0 填充**（zero-padded）。

所以，Redis 会：
- 把原来的 1 字节扩展到至少 2 字节（16 位）。
- 原来是：`00110001`（字符 '1'）
- 扩展后变成：`00110001 00000000`（后面补一个字节的 0）
- 然后设置第 10 位为 1：
  - 第 8 位是第一个新字节的第 0 位
  - 第 9 位是第 1 位
  - 第 10 位是第 2 位 → 把它设为 1

结果变成：
```
00110001 00000100
```

> 🔥 注意：原来的字符 `'1'` 已经被破坏了！现在这个字符串不再是可读的文本。

---

### ✅ 所以关键结论：

| 问题 | 回答 |
|------|------|
| 如果字符串不够长，能操作高位吗？ | ✅ **能！Redis 会自动扩展** |
| 扩展后空位填什么？ | ✅ **全部填 0**（zero-padded） |
| 原来的数据会被破坏吗？ | ✅ **会！** 一旦你用 `SETBIT` 操作高位，原来的字符串语义就没了 |
| 所以 Bitmap 和普通字符串能混用吗？ | ❌ **不建议！** 要么当普通字符串用，要么当 Bitmap 用，别混 |

---

### 💡 正确使用建议

- **不要**先 `SET mykey 1`，再用 `SETBIT` 操作高位。
- **应该**直接把某个 key 当作“纯位图”来用，不要赋予它文本含义。

比如：

```bash
# 直接当作位图用，没问题
SETBIT user:login:20240801 100 1
SETBIT user:login:20240801 1000000 1
```

Redis 会自动分配内存，从全 0 开始，你想设哪位就设哪位。

---

### 📌 再回答你的原问题：

> “如果这个 key 的字符串没有那么多位呢，比如 key 就是一个 1”

👉 Redis 会**自动扩展字符串长度**，用 0 填充中间空缺的位，让你可以操作任意高位（只要不超过 2³²）。

但代价是：**原来的字符串内容会被破坏**，因为它现在是一个“位数组”了，不再是文本。

---

### ✅ 类比理解

想象你有一个小盒子（1 字节），现在你要在第 10 个格子放东西，盒子不够大。

Redis 的做法是：
1. 拿出一个更大的盒子（自动扩展）
2. 中间空格都放上 0
3. 在第 10 个位置放上你要的 1

这就是 Redis Bitmap 的“动态扩展”能力，也是它强大又灵活的原因。

---

✅ 总结一句话：

> **Redis Bitmap 支持“稀疏”操作：即使字符串原来很短，你也可以直接操作很高的位偏移，Redis 会自动扩展并用 0 填充中间空位。**




## Redis Bitmap（位图）虽然简单，只支持 0 和 1 的状态，但因其**极致的空间效率**和**极快的统计能力**，在很多场景中堪称“神器”。以下是几个 **经典、实用、高频使用** 的场景，通俗易懂地为你讲解：

---

### 1. 用户签到系统（最常见）

**需求**：记录每个用户每天是否签到，支持：
- 查看某用户今天是否签到
- 统计今天有多少人签到
- 查看用户连续签到天数

**实现方式**：
- 每天一个 key：`sign:20240801`
- 用户 ID 作为 offset
- `SETBIT sign:20240801 1001 1` → 用户 ID 1001 今天签到了

**优势**：
- 1亿用户，每天只占 **12.5MB** 内存（1亿 bit ≈ 12.5MB）
- 统计签到人数用 `BITCOUNT`，**秒级返回**

---

###  2. 活跃用户统计（DAU/MAU）

**需求**：统计日活跃用户（DAU）、月活跃用户（MAU）

**实现方式**：
- 每天一个 key：`dau:20240801`
- 用户登录时：`SETBIT dau:20240801 <user_id> 1`

**统计**：
```bash
# 今天有多少人登录？
BITCOUNT dau:20240801

# 本月总共多少人活跃过？（OR 合并）
BITOP OR mau:202408 dau:20240801 dau:20240802 ... dau:20240831
BITCOUNT mau:202408
```

**优势**：
- 比数据库 `COUNT(DISTINCT user_id)` 快几十倍
- 存储成本极低

---

### 3. 用户行为标记（是否做过某事）

**需求**：记录用户是否：
- 阅读过某篇文章
- 参与过某个活动
- 领取过某个奖励

**实现方式**：
- `article:read:1001` → 记录谁读过文章 ID 1001
- `SETBIT article:read:1001 5001 1` → 用户 5001 读过了

**查询**：
- `GETBIT article:read:1001 5001` → 是否读过？

**优势**：
- 实现“去重”逻辑极其简单
- 适合做“已读/未读”状态管理

---

### 4. 布隆过滤器的底层支持（去重）

**需求**：快速判断一个 URL 是否已被爬取、一个用户是否已注册

**原理**：
- 布隆过滤器（Bloom Filter）内部就是多个 Bitmap
- 通过多个哈希函数映射到位图的不同位置

**优势**：
- 空间比 HashSet 小几十倍
- 查询速度极快（可能存在极低误判率）

> Redis 的 `RedisBloom` 模块就是基于 Bitmap 实现的。

---

### 5. 实时在线用户统计

**需求**：统计当前有多少用户在线

**实现方式**：
- 用用户 ID 作为 offset
- 上线时：`SETBIT online:users <user_id> 1`
- 下线时：`SETBIT online:users <user_id> 0`

**统计**：
```bash
BITCOUNT online:users  # 当前在线人数
```

**注意**：
- 要配合过期机制或定时清理
- 适合用户量大但并发不是极端高的场景

---

### 6. 权限或功能开关（小范围）

**需求**：控制某些用户是否开通某个高级功能

**实现方式**：
- `feature:premium` → 高级功能白名单
- `SETBIT feature:premium 1001 1` → 用户 1001 开通

**查询**：
- `GETBIT feature:premium 1001` → 是否有权限？

**优势**：
- 比数据库查权限表快
- 适合“开关类”布尔权限

---

### ✅ 总结：Bitmap 适合什么场景？

| 特点 | 是否适合 Bitmap |
|------|----------------|
| 数据是“是/否”、“有/无”、“在/不在” | ✅ 极其适合 |
| 用户量大（百万、千万、亿级） | ✅ 空间优势巨大 |
| 需要快速统计总数（如 DAU） | ✅ `BITCOUNT` 秒级返回 |
| 需要集合运算（交集、并集） | ✅ `BITOP` 一行搞定 |
| 数据需要频繁更新 | ✅ 单 bit 操作非常快 |
| 需要精确查询某个用户状态 | ✅ `GETBIT` O(1) 时间 |
