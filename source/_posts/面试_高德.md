## 2. Redis为什么这么快

**核心原因：**

1. **内存存储**：数据存储在内存中，读写速度比磁盘快几个数量级
2. **单线程模型**：避免了多线程竞争和锁开销
3. **高效的数据结构**：专门优化的数据结构（SDS、跳跃表、压缩列表等）
4. **I/O多路复用**：使用epoll等机制处理大量连接
5. **编码优化**：针对不同数据大小使用不同编码方式

**代码示例：**
```java
// Redis单线程模型示例
public class RedisSingleThread {
    public static void main(String[] args) {
        // Redis使用单线程处理命令，避免了锁竞争
        Jedis jedis = new Jedis("localhost", 6379);
        
        // 这些操作在Redis内部是串行执行的
        jedis.set("key1", "value1");
        jedis.set("key2", "value2");
        jedis.get("key1");
        
        // 批量操作提高效率
        Pipeline pipeline = jedis.pipelined();
        for (int i = 0; i < 1000; i++) {
            pipeline.set("key" + i, "value" + i);
        }
        pipeline.sync();
    }
}
```

## 3. 你知道Redis为什么要用Reactor模型吗？你项目中还有哪些组件也用了类似的设计

Reactor 模型就是一个“事件分发器”：它像服务员一样，统一接收所有客户的请求，然后派给工人去做，自己继续接待下一位，从而实现高并发、高性能的网络服务。

**Reactor模型优势：**
- **高并发处理**：单线程处理多个连接
- **事件驱动**：基于事件回调，非阻塞
- **资源高效利用**：避免为每个连接创建线程

**项目中类似设计：**
- **Netty**：基于Reactor模式的网络框架
- **Nginx**：事件驱动的Web服务器
- **Node.js**：单线程事件循环模型

**代码示例：**
```java
// Netty Reactor模式示例
public class NettyReactorExample {
    public static void main(String[] args) {
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);
        EventLoopGroup workerGroup = new NioEventLoopGroup();
        
        try {
            ServerBootstrap bootstrap = new ServerBootstrap();
            bootstrap.group(bossGroup, workerGroup)
                    .channel(NioServerSocketChannel.class)
                    .childHandler(new ChannelInitializer<SocketChannel>() {
                        @Override
                        protected void initChannel(SocketChannel ch) {
                            ch.pipeline().addLast(new EchoServerHandler());
                        }
                    });
            
            ChannelFuture f = bootstrap.bind(8080).sync();
            f.channel().closeFuture().sync();
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}
```

## 4. 讲讲io多路复用吧

**I/O多路复用概念：**
- 一个进程/线程可以监听多个文件描述符
- 当某个文件描述符就绪时，通知程序进行相应操作
- 避免了为每个连接创建线程的开销

**三种实现方式：**

1. **select**：跨平台，但效率较低
2. **poll**：改进select，但仍有性能问题
3. **epoll**：Linux特有，效率最高

**代码示例：**
```java
// Java NIO多路复用示例
public class NIOMultiplexingExample {
    public static void main(String[] args) throws IOException {
        Selector selector = Selector.open();
        ServerSocketChannel serverSocket = ServerSocketChannel.open();
        serverSocket.bind(new InetSocketAddress(8080));
        serverSocket.configureBlocking(false);
        serverSocket.register(selector, SelectionKey.OP_ACCEPT);
        
        while (true) {
            // 阻塞等待事件
            selector.select();
            Set<SelectionKey> keys = selector.selectedKeys();
            Iterator<SelectionKey> iterator = keys.iterator();
            
            while (iterator.hasNext()) {
                SelectionKey key = iterator.next();
                iterator.remove();
                
                if (key.isAcceptable()) {
                    // 处理连接事件
                    ServerSocketChannel server = (ServerSocketChannel) key.channel();
                    SocketChannel client = server.accept();
                    client.configureBlocking(false);
                    client.register(selector, SelectionKey.OP_READ);
                } else if (key.isReadable()) {
                    // 处理读事件
                    SocketChannel client = (SocketChannel) key.channel();
                    ByteBuffer buffer = ByteBuffer.allocate(1024);
                    client.read(buffer);
                    buffer.flip();
                    client.write(buffer);
                }
            }
        }
    }
}
```

## 7. 如果一个topic有5个queue，那我起10个consumer会怎么样？都能用上吗

**答案：不能都用上**

**原因分析：**
- RocketMQ的负载均衡策略是：一个Queue只能被一个Consumer消费
- 5个Queue最多只能被5个Consumer消费
- 剩余的5个Consumer会处于空闲状态

**负载均衡策略：**
```java
// 负载均衡示例
public class LoadBalanceExample {
    public static void main(String[] args) {
        // 假设有5个Queue，10个Consumer
        int queueCount = 5;
        int consumerCount = 10;
        
        // 计算每个Consumer分配的Queue数量
        int queuesPerConsumer = queueCount / consumerCount; // 0
        int remainder = queueCount % consumerCount; // 5
        
        // 前5个Consumer各分配1个Queue，后5个Consumer分配0个Queue
        for (int i = 0; i < consumerCount; i++) {
            if (i < remainder) {
                System.out.println("Consumer " + i + " 分配 " + (queuesPerConsumer + 1) + " 个Queue");
            } else {
                System.out.println("Consumer " + i + " 分配 " + queuesPerConsumer + " 个Queue");
            }
        }
    }
}
```

## 8. 什么是事务消息？RocketMQ怎么保证事务消息

**事务消息概念：**
- 保证本地事务和消息发送的原子性
- 解决分布式事务问题

**实现机制：**

1. **半事务消息**：先发送消息，但不立即投递
2. **本地事务执行**：执行本地业务逻辑
3. **事务状态回查**：根据本地事务结果决定消息是否投递

**代码示例：**
```java
// 事务消息完整示例
@Component
public class OrderTransactionService {
    
    @Autowired
    private TransactionMQProducer producer;
    
    @Autowired
    private OrderService orderService;
    
    public void createOrderWithTransaction(OrderRequest request) {
        // 1. 发送半事务消息
        Message msg = new Message("order-topic", 
            JSON.toJSONString(request).getBytes());
        
        try {
            TransactionSendResult result = producer.sendMessageInTransaction(msg, 
                new OrderTransactionListener());
            
            if (result.getLocalTransactionState() == LocalTransactionState.COMMIT_MESSAGE) {
                System.out.println("订单创建成功，消息已提交");
            } else {
                System.out.println("订单创建失败，消息已回滚");
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

class OrderTransactionListener implements TransactionListener {
    
    @Autowired
    private OrderService orderService;
    
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            // 解析消息
            OrderRequest request = JSON.parseObject(new String(msg.getBody()), 
                OrderRequest.class);
            
            // 执行本地事务
            Order order = orderService.createOrder(request);
            
            if (order != null) {
                return LocalTransactionState.COMMIT_MESSAGE;
            } else {
                return LocalTransactionState.ROLLBACK_MESSAGE;
            }
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        try {
            // 根据消息中的订单ID查询订单状态
            OrderRequest request = JSON.parseObject(new String(msg.getBody()), 
                OrderRequest.class);
            
            Order order = orderService.getOrderById(request.getOrderId());
            
            if (order != null && order.getStatus() == OrderStatus.CREATED) {
                return LocalTransactionState.COMMIT_MESSAGE;
            } else {
                return LocalTransactionState.ROLLBACK_MESSAGE;
            }
        } catch (Exception e) {
            return LocalTransactionState.UNKNOW;
        }
    }
}
```

## 9. 如果本地事务成功了，但是commit消息时因为网络问题失败了，RocketMQ怎么处理？是谁去查谁

**处理机制：**

1. **Broker主动回查**：Broker会定期回查Producer的本地事务状态
2. **Producer被动响应**：Producer需要实现回查接口
3. **最终一致性**：通过回查机制保证消息最终投递

**回查流程：**
```java
// 回查机制示例
public class TransactionCheckExample {
    
    public static void main(String[] args) {
        // Broker回查Producer的流程
        // 1. Broker发现半事务消息
        // 2. 调用Producer的checkLocalTransaction方法
        // 3. Producer查询本地事务状态
        // 4. 返回COMMIT/ROLLBACK/UNKNOW
        // 5. Broker根据结果决定是否投递消息
    }
    
    // Producer需要实现的回查接口
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 根据业务ID查询本地事务状态
        String orderId = extractOrderId(msg);
        Order order = orderService.getOrderById(orderId);
        
        if (order != null && order.getStatus() == OrderStatus.CREATED) {
            return LocalTransactionState.COMMIT_MESSAGE;
        } else if (order != null && order.getStatus() == OrderStatus.CANCELLED) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        } else {
            return LocalTransactionState.UNKNOW;
        }
    }
}
```

## 10. 什么是死信队列？和重试队列的区别？

**死信队列（DLQ）：**
- 消息经过多次重试后仍然失败，进入死信队列
- 用于存储无法正常处理的消息
- 需要人工干预处理

**重试队列：**
- 消息消费失败时的临时存储队列
- 系统会自动重试消费
- 重试次数达到上限后进入死信队列

**代码示例：**
```java
// 死信队列和重试队列示例
@Component
public class DeadLetterQueueExample {
    
    @Autowired
    private DefaultMQPushConsumer consumer;
    
    public void setupConsumer() {
        // 设置重试次数
        consumer.setMaxReconsumeTimes(3);
        
        // 注册消息监听器
        consumer.registerMessageListener(new MessageListenerConcurrently() {
            @Override
            public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs, 
                    ConsumeConcurrentlyContext context) {
                
                for (MessageExt msg : msgs) {
                    try {
                        // 处理消息
                        processMessage(msg);
                        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
                    } catch (Exception e) {
                        // 消息处理失败，会进入重试队列
                        log.error("消息处理失败，将进行重试", e);
                        
                        // 检查重试次数
                        if (msg.getReconsumeTimes() >= 3) {
                            // 重试次数达到上限，消息将进入死信队列
                            log.error("消息重试次数达到上限，进入死信队列");
                        }
                        
                        return ConsumeConcurrentlyStatus.RECONSUME_LATER;
                    }
                }
                return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
            }
        });
    }
    
    // 死信队列消费者
    @Component
    public class DeadLetterConsumer {
        
        @Autowired
        private DefaultMQPushConsumer dlqConsumer;
        
        public void setupDeadLetterConsumer() {
            // 订阅死信队列
            dlqConsumer.subscribe("%DLQ%order-topic", "*");
            
            dlqConsumer.registerMessageListener(new MessageListenerConcurrently() {
                @Override
                public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs, 
                        ConsumeConcurrentlyContext context) {
                    
                    for (MessageExt msg : msgs) {
                        // 处理死信消息
                        handleDeadLetter(msg);
                    }
                    return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
                }
            });
        }
        
        private void handleDeadLetter(MessageExt msg) {
            // 记录死信消息
            log.error("收到死信消息: {}", new String(msg.getBody()));
            
            // 可以发送告警、记录日志、人工处理等
            sendAlert("发现死信消息，需要人工处理");
        }
    }
}
```

## 12. 你put一条数据进 ES，立马用 match 查不到，但term 可以查到，问题在哪

**问题原因：**
- **match查询**：会对查询词进行分词，然后查询倒排索引
- **term查询**：精确匹配，不进行分词
- **近实时性**：ES默认1秒刷新一次，新数据可能还未建立倒排索引

**代码示例：**
```java
public class MatchVsTermExample {
    
    @Autowired
    private RestHighLevelClient client;
    
    public void testMatchVsTerm() throws IOException {
        // 1. 写入数据
        IndexRequest request = new IndexRequest("test")
            .id("1")
            .source(Map.of("content", "hello world"));
        
        client.index(request, RequestOptions.DEFAULT);
        
        // 2. 立即查询
        // match查询 - 可能查不到（需要分词，依赖倒排索引）
        SearchRequest matchRequest = new SearchRequest("test");
        matchRequest.source(SearchSourceBuilder.searchSource()
            .query(QueryBuilders.matchQuery("content", "hello")));
        
        SearchResponse matchResponse = client.search(matchRequest, RequestOptions.DEFAULT);
        System.out.println("Match查询结果: " + matchResponse.getHits().getTotalHits());
        
        // term查询 - 能查到（精确匹配，不依赖倒排索引）
        SearchRequest termRequest = new SearchRequest("test");
        termRequest.source(SearchSourceBuilder.searchSource()
            .query(QueryBuilders.termQuery("content", "hello world")));
        
        SearchResponse termResponse = client.search(termRequest, RequestOptions.DEFAULT);
        System.out.println("Term查询结果: " + termResponse.getHits().getTotalHits());
        
        // 3. 强制刷新后再次查询
        RefreshRequest refreshRequest = new RefreshRequest("test");
        client.indices().refresh(refreshRequest, RequestOptions.DEFAULT);
        
        SearchResponse matchResponseAfterRefresh = client.search(matchRequest, RequestOptions.DEFAULT);
        System.out.println("刷新后Match查询结果: " + matchResponseAfterRefresh.getHits().getTotalHits());
    }
}
```

## 13. 假如我用Redis做了一个排行榜，基于 ZSet 维护，某天发现榜单排名混乱且跳动频繁，你觉得可能是哪几类问题？

**可能的问题：**

1. **并发更新问题**：多个客户端同时更新分数
2. **网络延迟**：Redis集群节点间同步延迟

## 17. 手撕：设计一个map，满足以下几点: 字符串表示键，整数表示值返回具有前缀等于给定字符串的键的值的总和

**解题思路：**
- 使用Trie树（前缀树）存储字符串键
- 每个节点记录以该前缀结尾的所有值的总和
- 支持快速前缀查询和值更新

**代码实现：**
```java
// Trie树节点
class TrieNode {
    Map<Character, TrieNode> children;
    int value; // 以当前节点结尾的键的值
    int prefixSum; // 以当前节点为前缀的所有值的总和
    
    public TrieNode() {
        this.children = new HashMap<>();
        this.value = 0;
        this.prefixSum = 0;
    }
}

// 前缀Map实现
public class PrefixMap {
    private TrieNode root;
    
    public PrefixMap() {
        this.root = new TrieNode();
    }
    
    // 插入或更新键值对
    public void put(String key, int value) {
        if (key == null) return;
        
        TrieNode current = root;
        int oldValue = get(key); // 获取旧值
        
        // 遍历字符串，构建或更新Trie树
        for (char c : key.toCharArray()) {
            current.children.putIfAbsent(c, new TrieNode());
            current = current.children.get(c);
            
            // 更新前缀和：减去旧值，加上新值
            current.prefixSum = current.prefixSum - oldValue + value;
        }
        
        // 设置当前节点的值
        current.value = value;
    }
    
    // 获取指定键的值
    public int get(String key) {
        if (key == null) return 0;
        
        TrieNode current = root;
        for (char c : key.toCharArray()) {
            if (!current.children.containsKey(c)) {
                return 0;
            }
            current = current.children.get(c);
        }
        return current.value;
    }
    
    // 获取指定前缀的所有值的总和
    public int getPrefixSum(String prefix) {
        if (prefix == null) return 0;
        
        TrieNode current = root;
        for (char c : prefix.toCharArray()) {
            if (!current.children.containsKey(c)) {
                return 0;
            }
            current = current.children.get(c);
        }
        return current.prefixSum;
    }
    
    // 删除指定键
    public void remove(String key) {
        if (key == null) return;
        
        int value = get(key);
        if (value == 0) return; // 键不存在
        
        TrieNode current = root;
        for (char c : key.toCharArray()) {
            if (!current.children.containsKey(c)) {
                return;
            }
            current = current.children.get(c);
            current.prefixSum -= value; // 更新前缀和
