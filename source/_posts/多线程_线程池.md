## 线程池的意义与优势

- **资源复用**：线程池通过复用已创建的线程，避免频繁创建/销毁线程带来的性能损耗。
- **统一管理**：集中管理线程生命周期，便于监控和调优。
- **任务调度**：支持任务排队、定时/周期执行、批量处理等高级功能。
- **防止资源耗尽**：通过限制最大线程数，防止系统因过多线程而崩溃。
- **提升响应速度**：任务无需等待新线程创建，直接复用空闲线程。

## Executor 框架概览

Java 的 `Executor` 框架（`java.util.concurrent` 包）实现了任务提交与执行的解耦，极大简化了并发编程。其核心思想是：
- **提交任务**（Runnable/Callable）
- **由线程池统一调度和执行**

---

## 核心接口与实现类

### 1. Executor
最基础接口，定义了：
```java
void execute(Runnable command);
```
用于提交任务，但不关心执行细节。

### 2. ExecutorService
扩展了 Executor，支持：
- 任务提交（submit）
- 任务取消、结果获取（Future）
- 批量任务处理（invokeAll/invokeAny）
- 线程池关闭与状态管理

### 3. ScheduledExecutorService
支持定时/周期性任务调度。

### 4. 主要实现类
- **ThreadPoolExecutor**：最常用的线程池实现，支持灵活参数配置。
- **ScheduledThreadPoolExecutor**：支持定时/周期任务。
- **ForkJoinPool**：适合递归分治、并行计算场景。

## 线程池的创建与使用

### 1. Executors 工具类
`Executors` 提供多种便捷工厂方法：

| 方法 | 说明 |
|------|------|
| newFixedThreadPool(n) | 固定大小线程池 |
| newSingleThreadExecutor() | 单线程池，顺序执行 |
| newCachedThreadPool() | 可伸缩线程池，适合大量短任务 |
| newScheduledThreadPool(n) | 定时/周期任务线程池 |
| newWorkStealingPool() | 工作窃取线程池（Java 8+） |

**示例：**
```java
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(() -> System.out.println("Hello from thread pool"));
pool.shutdown();
```

> ⚠️ 建议生产环境优先使用 ThreadPoolExecutor 显式配置参数，避免 Executors 默认策略带来的风险（如 OOM）。

### 2. 线程池的基本用法
- 提交任务（execute/submit）
- 获取结果（Future）
- 关闭线程池（shutdown/shutdownNow）

**示例：**
```java
Future<String> future = pool.submit(() -> "result");
System.out.println(future.get());
pool.shutdown();
```

## ThreadPoolExecutor 深入解析

`ThreadPoolExecutor` 是线程池的核心实现，支持高度自定义。

### 1. 构造参数
```java
ThreadPoolExecutor(
    int corePoolSize,         // 核心线程数
    int maximumPoolSize,      // 最大线程数
    long keepAliveTime,       // 非核心线程空闲存活时间
    TimeUnit unit,            // 时间单位
    BlockingQueue<Runnable> workQueue, // 任务队列
    ThreadFactory threadFactory,       // 线程工厂
    RejectedExecutionHandler handler   // 拒绝策略
)
```

### 2. 线程池运行流程
1. 任务提交到线程池
2. 若运行线程数 < corePoolSize，直接创建新线程执行
3. 否则，任务进入队列等待
4. 若队列满且线程数 < maximumPoolSize，创建新线程执行
5. 若队列满且线程数已达最大，触发拒绝策略

### 3. 任务队列类型
- **ArrayBlockingQueue**：有界数组队列，常用
- **LinkedBlockingQueue**：无界链表队列，默认
- **SynchronousQueue**：不存储任务，直接交给线程
- **PriorityBlockingQueue**：优先级队列

### 4. 拒绝策略
- **AbortPolicy**（默认）：抛出异常
- **CallerRunsPolicy**：由提交任务的线程执行
- **DiscardPolicy**：直接丢弃
- **DiscardOldestPolicy**：丢弃队列最老任务，尝试执行新任务

### 5. 线程池监控与管理
- `getPoolSize()`：当前线程数
- `getActiveCount()`：活跃线程数
- `getCompletedTaskCount()`：已完成任务数
- `prestartAllCoreThreads()`：预启动所有核心线程

**示例：**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, 4, 5, TimeUnit.SECONDS, new LinkedBlockingQueue<>());
for (int i = 0; i < 5; i++) {
    int taskNum = i + 1;
    executor.execute(() -> {
        System.out.println("Task " + taskNum + " in " + Thread.currentThread().getName());
    });
}
executor.shutdown();
```

## 线程池参数调优建议

- **核心线程数（corePoolSize）**：CPU 密集型 ≈ CPU 核心数；IO 密集型 ≈ 2×CPU 核心数
- **最大线程数（maximumPoolSize）**：根据业务并发量和硬件资源合理设置
- **队列类型与长度**：有界队列防止 OOM，无界队列需谨慎
- **拒绝策略**：根据业务容忍度选择，推荐自定义日志/报警
- **线程工厂**：自定义线程名便于排查问题
- **监控与报警**：定期监控线程池状态，及时发现异常

/**
 * 拒绝策略接口
 */
interface RejectionPolicy {
    /**
     * 处理被拒绝的任务
     * @param task 被拒绝的任务
     * @param threadPool 线程池实例
     */
    void rejectedExecution(Runnable task, ThreadPool threadPool);
}

/**
 * 默认拒绝策略：抛出异常
 */
class AbortPolicy implements RejectionPolicy {
    @Override
    public void rejectedExecution(Runnable task, ThreadPool threadPool) {
        throw new RejectedExecutionException("任务被拒绝执行，线程池已满");
    }
}

/**
 * 调用者运行策略：由提交任务的线程执行
 */
class CallerRunsPolicy implements RejectionPolicy {
    @Override
    public void rejectedExecution(Runnable task, ThreadPool threadPool) {
        System.out.println("任务被拒绝，由调用者线程执行: " + Thread.currentThread().getName());
        if (!threadPool.isShutdown()) {
            task.run();
        }
    }
}

/**
 * 直接丢弃策略：静默丢弃任务
 */
class DiscardPolicy implements RejectionPolicy {
    @Override
    public void rejectedExecution(Runnable task, ThreadPool threadPool) {
        System.out.println("任务被直接丢弃");
        // 什么都不做，直接丢弃
    }
}

/**
 * 丢弃最老任务策略：丢弃队列中最老的任务，然后尝试执行新任务
 */
class DiscardOldestPolicy implements RejectionPolicy {
    @Override
    public void rejectedExecution(Runnable task, ThreadPool threadPool) {
        System.out.println("尝试丢弃最老任务并执行新任务");
        if (!threadPool.isShutdown()) {
            // 尝试从队列中移除最老的任务
            Runnable discardedTask = threadPool.getWorkQueue().poll();
            if (discardedTask != null) {
                System.out.println("丢弃了最老的任务");
            }
            // 尝试将新任务放入队列
            try {
                threadPool.getWorkQueue().put(task);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                System.out.println("放入新任务时被中断");
            }
        }
    }
}

    public RejectedExecutionException(String message, Throwable cause) {

/**
 * 自定义拒绝策略：重试策略
 */
class RetryPolicy implements RejectionPolicy {
    private final int maxRetries;
    private final long retryDelayMs;
    
    public RetryPolicy(int maxRetries, long retryDelayMs) {
        this.maxRetries = maxRetries;
        this.retryDelayMs = retryDelayMs;
    }
    
    @Override
    public void rejectedExecution(Runnable task, ThreadPool threadPool) {
        System.out.println("任务被拒绝，尝试重试...");
        
        for (int i = 0; i < maxRetries; i++) {
            try {
                Thread.sleep(retryDelayMs);
                
                // 重新尝试提交任务
                if (threadPool.getWorkQueue().offer(task)) {
                    System.out.println("重试成功，任务已重新加入队列");
                    return;
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
        }
        
        // 重试失败，抛出异常
        throw new RejectedExecutionException("任务重试" + maxRetries + "次后仍然被拒绝");
    }
}

/**
 * 自定义拒绝策略：降级策略
 */
class FallbackPolicy implements RejectionPolicy {
    private final Runnable fallbackTask;
    
    public FallbackPolicy(Runnable fallbackTask) {
        this.fallbackTask = fallbackTask;
    }
    
    @Override
    public void rejectedExecution(Runnable task, ThreadPool threadPool) {
        System.out.println("任务被拒绝，执行降级策略");
        
        try {
            // 执行降级任务
            if (fallbackTask != null) {
                fallbackTask.run();
            }
        } catch (Exception e) {
            System.err.println("降级任务执行失败: " + e.getMessage());
        }
    }
}

class ThreadPool {
    private final BlockingQueue<Runnable> workQueue;
    private final Set<Worker> workers;
    private final int coreSize;
    private final long timeout;
    private final TimeUnit timeUnit;
    private final AtomicBoolean isShutdown;
    private final Lock mainLock = new ReentrantLock();
    private final RejectionPolicy rejectionPolicy;

    public ThreadPool(int coreSize, long timeout, TimeUnit timeUnit, int queueCapacity) {
        this(coreSize, timeout, timeUnit, queueCapacity, new AbortPolicy());
    }

    public ThreadPool(int coreSize, long timeout, TimeUnit timeUnit, int queueCapacity, RejectionPolicy rejectionPolicy) {
        if (coreSize <= 0 || timeout <= 0 || queueCapacity <= 0) {
            throw new IllegalArgumentException("参数必须大于0");
        }
        if (rejectionPolicy == null) {
            throw new NullPointerException("拒绝策略不能为null");
        }
        
        this.coreSize = coreSize;
        this.timeout = timeout;
        this.timeUnit = timeUnit;
        this.workQueue = new BlockingQueue<>(queueCapacity);
        this.workers = new HashSet<>();
        this.isShutdown = new AtomicBoolean(false);
        this.rejectionPolicy = rejectionPolicy;
    }

    /**
     * 执行任务
     */
    public void execute(Runnable runnable) {
        if (runnable == null) {
            throw new NullPointerException("任务不能为null");
        }
        
        if (isShutdown.get()) {
            throw new IllegalStateException("线程池已关闭");
        }

        mainLock.lock();
        try {
            // 如果当前线程数小于核心线程数，创建新线程
            if (workers.size() < coreSize) {
                System.out.println("新增worker线程，当前worker数量: " + workers.size());
                Worker worker = new Worker(runnable);
                workers.add(worker);
                worker.start();
            } else {
                // 尝试将任务放入队列
                if (!workQueue.offer(runnable)) {
                    // 队列已满，触发拒绝策略
                    System.out.println("队列已满，触发拒绝策略");
                    rejectionPolicy.rejectedExecution(runnable, this);
                } else {
                    System.out.println("任务加入等待队列，当前队列大小: " + workQueue.size());
                }
            }
        } finally {
            mainLock.unlock();
        }
    }

    /**
     * 关闭线程池
     */
    public void shutdown() {
        if (isShutdown.compareAndSet(false, true)) {
            System.out.println("正在关闭线程池...");
            mainLock.lock();
            try {
                for (Worker worker : workers) {
                    worker.interrupt();
                }
            } finally {
                mainLock.unlock();
            }
        }
    }

    /**
     * 获取当前活跃线程数
     */
    public int getActiveCount() {
        mainLock.lock();
        try {
            return workers.size();
        } finally {
            mainLock.unlock();
        }
    }

    /**
     * 获取队列中的任务数
     */
    public int getQueueSize() {
        return workQueue.size();
    }

    /**
     * 检查线程池是否已关闭
     */
    public boolean isShutdown() {
        return isShutdown.get();
    }

    /**
     * 获取工作队列（用于拒绝策略）
     */
    BlockingQueue<Runnable> getWorkQueue() {
        return workQueue;
    }

    /**
     * Worker线程类
     */
    private class Worker extends Thread {
        private Runnable firstTask;
        private volatile boolean completedTasks = false;

        public Worker(Runnable firstTask) {
            this.firstTask = firstTask;
            setName("Worker-" + getId());
        }

        @Override
        public void run() {
            try {
                // 执行第一个任务
                if (firstTask != null) {
                    runTask(firstTask);
                }
                
                // 循环执行队列中的任务
                while (!isShutdown.get() || !workQueue.isEmpty()) {
                    Runnable task = workQueue.poll(timeout, timeUnit);
                    if (task != null) {
                        runTask(task);
                        completedTasks = true;
                    } else if (isShutdown.get()) {
                        // 线程池已关闭且队列为空，退出循环
                        break;
                    }
                }
            } catch (InterruptedException e) {
                // 线程被中断，正常退出
                Thread.currentThread().interrupt();
            } finally {
                // 从workers集合中移除自己
                mainLock.lock();
                try {
                    workers.remove(this);
                    System.out.println("Worker线程 " + getName() + " 退出，当前活跃线程数: " + workers.size());
                } finally {
                    mainLock.unlock();
                }
            }
        }

        private void runTask(Runnable task) {
            try {
                task.run();
            } catch (Exception e) {
                System.err.println("任务执行异常: " + e.getMessage());
                e.printStackTrace();
            }
        }
    }
}

/**
 * 线程安全的阻塞队列
 */
class BlockingQueue<T> {
    private final Deque<T> queue = new ArrayDeque<>();
    private final Lock lock = new ReentrantLock();
    private final Condition notEmpty = lock.newCondition();
    private final Condition notFull = lock.newCondition();
    private final int capacity;

    public BlockingQueue(int capacity) {
        if (capacity <= 0) {
            throw new IllegalArgumentException("容量必须大于0");
        }
        this.capacity = capacity;
    }

    /**
     * 带超时的获取元素
     */
    public T poll(long timeout, TimeUnit unit) throws InterruptedException {
        long nanos = unit.toNanos(timeout);
        lock.lockInterruptibly();
        try {
            while (queue.isEmpty()) {
                if (nanos <= 0) {
                    return null;
                }
                nanos = notEmpty.awaitNanos(nanos);
            }
            T element = queue.removeFirst();
            notFull.signal();
            return element;
        } finally {
            lock.unlock();
        }
    }

    /**
     * 阻塞获取元素
     */
    public T take() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (queue.isEmpty()) {
                notEmpty.await();
            }
            T element = queue.removeFirst();
            notFull.signal();
            return element;
        } finally {
            lock.unlock();
        }
    }

    /**
     * 阻塞添加元素
     */
    public void put(T element) throws InterruptedException {
        if (element == null) {
            throw new NullPointerException("元素不能为null");
        }
        
        lock.lockInterruptibly();
        try {
            while (queue.size() == capacity) {
                notFull.await();
            }
            queue.addLast(element);
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    /**
     * 非阻塞添加元素
     */
    public boolean offer(T element) {
        if (element == null) {
            throw new NullPointerException("元素不能为null");
        }
        
        lock.lock();
        try {
            if (queue.size() == capacity) {
                return false;
            }
            queue.addLast(element);
            notEmpty.signal();
            return true;
        } finally {
            lock.unlock();
        }
    }

    /**
     * 获取队列大小
     */
    public int size() {
        lock.lock();
        try {
            return queue.size();
        } finally {
            lock.unlock();
        }
    }

    /**
     * 检查队列是否为空
     */
    public boolean isEmpty() {
        lock.lock();
        try {
            return queue.isEmpty();
        } finally {
            lock.unlock();
        }
    }

    /**
     * 检查队列是否已满
     */
    public boolean isFull() {
        lock.lock();
        try {
            return queue.size() == capacity;
        } finally {
            lock.unlock();
        }
    }
}

## 拒绝策略详解

### 什么是拒绝策略？

拒绝策略是线程池中处理任务被拒绝执行时的策略。当线程池和任务队列都满时，新提交的任务会被拒绝，此时需要根据业务需求选择合适的处理方式。

### 内置拒绝策略

#### 1. AbortPolicy（默认策略）
- **行为**：抛出 `RejectedExecutionException` 异常
- **适用场景**：需要明确知道任务被拒绝的情况
- **优点**：能够及时发现线程池负载过高的问题
- **缺点**：可能导致调用方异常，需要额外的异常处理

#### 2. CallerRunsPolicy
- **行为**：由提交任务的线程执行被拒绝的任务
- **适用场景**：对任务执行时间不敏感，希望降低提交速度
- **优点**：能够自动调节提交速度，防止系统过载
- **缺点**：可能阻塞调用方线程

#### 3. DiscardPolicy
- **行为**：静默丢弃被拒绝的任务
- **适用场景**：任务可以被丢弃，不重要的任务
- **优点**：不会影响系统稳定性
- **缺点**：任务丢失，可能影响业务逻辑

#### 4. DiscardOldestPolicy
- **行为**：丢弃队列中最老的任务，然后尝试执行新任务
- **适用场景**：新任务比老任务更重要
- **优点**：保证新任务能够执行
- **缺点**：可能丢失重要的老任务

### 自定义拒绝策略

#### 1. 日志告警策略（LogAndAlertPolicy）
```java
// 使用示例
ThreadPool threadPool = new ThreadPool(2, 1000, TimeUnit.MILLISECONDS, 10, 
    new LogAndAlertPolicy("订单处理线程池"));
```
- **特点**：记录详细日志并发送告警
- **适用场景**：生产环境监控，需要及时发现问题

#### 2. 重试策略（RetryPolicy）
```java
// 使用示例
ThreadPool threadPool = new ThreadPool(2, 1000, TimeUnit.MILLISECONDS, 10, 
    new RetryPolicy(3, 1000)); // 重试3次，每次间隔1秒
```
- **特点**：自动重试被拒绝的任务
- **适用场景**：任务比较重要，可以接受延迟执行

#### 3. 降级策略（FallbackPolicy）
```java
// 使用示例
Runnable fallbackTask = () -> System.out.println("执行降级逻辑");
ThreadPool threadPool = new ThreadPool(2, 1000, TimeUnit.MILLISECONDS, 10, 
    new FallbackPolicy(fallbackTask));
```
- **特点**：执行降级任务替代被拒绝的任务
- **适用场景**：有备选方案，保证系统可用性

### 拒绝策略选择建议

#### 1. 根据业务重要性选择
- **关键业务**：使用 `CallerRunsPolicy` 或自定义重试策略
- **一般业务**：使用 `AbortPolicy` 或 `DiscardOldestPolicy`
- **非关键业务**：使用 `DiscardPolicy`

#### 2. 根据系统负载选择
- **高负载系统**：使用 `CallerRunsPolicy` 自动调节
- **稳定负载系统**：使用 `AbortPolicy` 及时发现问题
- **突发负载系统**：使用 `DiscardOldestPolicy` 保证新任务

#### 3. 根据监控需求选择
- **需要监控**：使用自定义日志告警策略
- **简单监控**：使用 `AbortPolicy` 配合异常监控
- **无需监控**：使用 `DiscardPolicy`

### 最佳实践

1. **生产环境推荐**：使用自定义日志告警策略，及时发现问题
2. **开发环境推荐**：使用 `AbortPolicy`，便于调试
3. **测试环境推荐**：使用 `CallerRunsPolicy`，模拟真实负载
4. **监控告警**：配合线程池监控，设置合理的告警阈值
5. **参数调优**：根据拒绝策略的执行情况，调整线程池参数

### 监控指标

- **拒绝次数**：统计任务被拒绝的次数
- **拒绝率**：拒绝次数 / 总提交次数
- **队列使用率**：当前队列大小 / 队列容量
- **线程使用率**：活跃线程数 / 最大线程数

通过这些指标可以及时发现线程池配置问题，进行相应的调优。

## Fork / Join

Fork/Join 框架是 Java 7 引入的一个并行编程框架，旨在帮助开发者更方便地编写高效、可扩展的并行程序。它基于“分治法”（Divide-and-Conquer）策略，将一个大任务递归分解为多个小任务并行执行，然后合并这些小任务的结果以得到最终结果。Fork/Join 框架特别适合于那些可以被分解成许多相似的小任务，并且这些小任务之间几乎没有依赖关系的情况。

### 核心组件

1. **ForkJoinPool**：这是 Fork/Join 框架的核心执行引擎，类似于线程池。它管理着工作窃取算法(work-stealing algorithm)下的工作线程集合。每个工作线程都维护着自己的双端队列(deque)，用于存储待执行的任务。

2. **ForkJoinTask<V>**：这是一个抽象类，代表了可以在 ForkJoinPool 中执行的任务。所有的 Fork/Join 任务都需要继承自这个类或者其子类。
   - **RecursiveAction**：适用于不需要返回结果的任务。
   - **RecursiveTask<V>**：适用于需要返回结果的任务。

3. **Work-Stealing Algorithm**：这是一种分配任务的方法，当某个线程完成了自己 deque 中的所有任务后，它可以尝试从其他线程的 deque 的尾部“偷取”任务来执行，从而提高了资源利用率和负载均衡。

### 工作流程

1. **Fork**：任务开始时，如果当前任务规模足够小，则直接计算；否则，将其分割(split)为两个或更多的子任务，并对每个子任务调用 `fork()` 方法异步执行。
   
2. **Join**：对于每个已经 fork 出去的子任务，父任务会调用 `join()` 方法等待子任务完成，并获取其结果。注意，`join()` 不会阻塞线程，除非当前线程没有其他任务可做。

3. **合并结果**：一旦所有子任务都完成了，父任务就会合并它们的结果，生成最终结果。

### 示例代码

这里有一个简单的例子，展示了如何使用 RecursiveTask 来计算数组中元素的总和：

```java
import java.util.concurrent.RecursiveTask;
import java.util.concurrent.ForkJoinPool;

public class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10; // 阈值，决定是否继续拆分任务
    private long[] array;
    private int start;
    private int end;

    public SumTask(long[] array, int start, int end) {
        this.array = array;
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            long sum = 0;
            for (int i = start; i < end; i++) {
                sum += array[i];
            }
            return sum;
        } else {
            int middle = (start + end) / 2;
            SumTask leftTask = new SumTask(array, start, middle);
            SumTask rightTask = new SumTask(array, middle, end);

            // 分别启动子任务
            leftTask.fork();
            rightTask.fork();

            // 合并结果
            return leftTask.join() + rightTask.join();
        }
    }

    public static void main(String[] args) {
        long[] numbers = { /* 初始化你的数据 */ };
        ForkJoinPool pool = new ForkJoinPool();
        SumTask task = new SumTask(numbers, 0, numbers.length);
        long result = pool.invoke(task);
        System.out.println("Sum is: " + result);
    }
}
```

### 特点与优势

- **高效性**：通过工作窃取算法有效地利用多核处理器的能力，减少了线程间的竞争。
- **灵活性**：可以轻松处理各种类型的并行问题，特别是那些可以通过分治法解决的问题。
- **易用性**：简化了并行程序的编写过程，使得开发人员无需手动管理线程同步等复杂问题。

### 1. 线程池的工作原理？

线程池的核心思想是**“池化”**：预先创建一组可重用的线程，避免频繁创建和销毁线程带来的性能开销。

**工作流程：**

1.  **提交任务**：客户端调用 `execute(Runnable command)` 或 `submit(Callable<T> task)` 提交任务。
2.  **判断核心线程**：
    *   如果**当前线程数 < 核心线程数（corePoolSize）**，即使有空闲线程，也会**创建新线程**来执行任务。
    *   如果**当前线程数 >= corePoolSize**，进入下一步。
3.  **尝试入队**：
    *   将任务放入**工作队列（workQueue）**。
    *   如果入队成功，任务在队列中等待。
    *   如果入队失败（队列已满），进入下一步。
4.  **判断最大线程数**：
    *   如果**当前线程数 < 最大线程数（maximumPoolSize）**，则**创建非核心线程**来执行这个新任务。
    *   如果**当前线程数 >= maximumPoolSize**，进入下一步。
5.  **执行拒绝策略**：
    *   触发**拒绝策略（RejectedExecutionHandler）**，如抛出异常、丢弃任务等。

**简单记忆：**
> **核心线程不够 → 入队 → 队列满 → 创建非核心线程 → 还不够 → 拒绝**

### 2. `ThreadPoolExecutor` 各参数含义及调优建议？

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    int corePoolSize,
    int maximumPoolSize,
    long keepAliveTime,
    TimeUnit unit,
    BlockingQueue<Runnable> workQueue,
    ThreadFactory threadFactory,
    RejectedExecutionHandler handler
);
```

| 参数 | 含义 | 调优建议 |
| :--- | :--- | :--- |
| **`corePoolSize`** | **核心线程数**。这些线程会一直存活（除非设置了 `allowCoreThreadTimeOut`）。 | - CPU 密集型任务：`N_cpu + 1`（N 为 CPU 核心数）<br>- I/O 密集型任务：`N_cpu * 2` 或更高（因为线程常阻塞） |
| **`maximumPoolSize`** | **最大线程数**。线程池允许创建的线程总数上限。 | - 通常设置为 `corePoolSize` 的 2-4 倍，用于应对突发流量。<br>- 需结合系统资源（CPU、内存、文件句柄等）设置。 |
| **`keepAliveTime`** | **非核心线程的空闲存活时间**。超过 `corePoolSize` 的线程，在空闲超过此时间后会被回收。 | - 一般设置为 30-60 秒。<br>- 如果希望核心线程也能回收，可设置 `allowCoreThreadTimeOut(true)`。 |
| **`unit`** | `keepAliveTime` 的时间单位（如 `SECONDS`, `MINUTES`）。 | - 根据业务需求选择。 |
| **`workQueue`** | **工作队列**。用于存放待执行的任务。 | - **推荐有界队列**（如 `ArrayBlockingQueue`），防止 OOM。<br>- 避免使用无界队列（如 `newFixedThreadPool` 默认的 `LinkedBlockingQueue`）。 |
| **`threadFactory`** | **线程工厂**。用于创建新线程。可以自定义线程名称、优先级、是否为守护线程等。 | - **强烈建议自定义**，为线程命名（如 `biz-pool-%d`），便于日志追踪和问题排查。 |
| **`handler`** | **拒绝策略**。当线程池无法处理新任务时的处理策略。 | - 根据业务选择合适的策略（见问题 3）。 |

### 3. 线程池的拒绝策略有哪些？适用场景？

| 拒绝策略 | 类名 | 行为 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **中止策略** | `AbortPolicy`（默认） | 抛出 `RejectedExecutionException` 异常。 | - 对数据一致性要求高，宁愿失败也不愿丢失或延迟。<br>- 系统设计应能处理此异常。 |
| **调用者运行策略** | `CallerRunsPolicy` | 由**提交任务的线程**（即调用者线程）自己来执行该任务。 | - 适用于**非关键任务**，可以接受任务执行变慢。<br>- 能起到“背压”作用，减缓任务提交速度。 |
| **丢弃策略** | `DiscardPolicy` | **静默丢弃**新提交的任务，不抛异常。 | - 适用于**可丢失任务**的场景，如日志收集、监控数据上报。 |
| **丢弃最旧策略** | `DiscardOldestPolicy` | 丢弃**队列中最老的一个任务**（即将被执行的那个），然后尝试提交新任务。 | - 适用于**实时性要求高**的场景，如消息推送，宁愿丢旧消息也不愿队列积压。 |

### 4. `Executors` 工厂方法的风险？为什么生产不推荐？

`Executors` 提供了便捷的工厂方法，但**在生产环境中应避免直接使用**，原因如下：

| 工厂方法 | 风险 | 详细说明 |
| :--- | :--- | :--- |
| `newFixedThreadPool(n)` | **使用无界队列** | 内部使用 `new LinkedBlockingQueue<>()`（容量 `Integer.MAX_VALUE`）。如果任务处理速度跟不上提交速度，队列会无限增长，最终导致 `OutOfMemoryError`。 |
| `newCachedThreadPool()` | **线程数无上限** | `maximumPoolSize = Integer.MAX_VALUE`。在高并发下可能创建海量线程，耗尽系统资源（CPU、内存、文件句柄等），导致系统崩溃。 |
| `newSingleThreadExecutor()` | **使用无界队列** | 同 `newFixedThreadPool`，存在 OOM 风险。 |

**✅ 正确做法**：**直接使用 `ThreadPoolExecutor` 构造函数**，显式指定所有参数，尤其是使用**有界队列**和**合理的线程数上限**。

### 5. 如何优雅关闭线程池？

目标是：**停止接收新任务，并等待已提交的任务执行完毕**。

```java
// 1. 停止接收新任务
executor.shutdown();

// 2. 等待任务完成（推荐设置超时）
try {
    if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
        // 超时后，强制关闭（中断正在执行的任务）
        executor.shutdownNow();
        // 再次等待，确保任务响应中断
        if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
            System.err.println("线程池未能正常关闭");
        }
    }
} catch (InterruptedException e) {
    // 当前线程被中断，也强制关闭
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
```

**关键方法**：
*   `shutdown()`：平滑关闭，不再接收新任务，但会执行完队列中的任务。
*   `shutdownNow()`：立即关闭，尝试中断所有正在执行的任务，并返回队列中未处理的任务。
*   `awaitTermination(timeout, unit)`：阻塞等待线程池关闭，或超时。

### 6. 线程池如何防止 OOM？

1.  **使用有界队列**：如 `ArrayBlockingQueue(capacity)`，明确限制队列大小。
2.  **合理设置线程数**：避免 `maximumPoolSize` 过大，防止创建过多线程耗尽内存。
3.  **配置合适的拒绝策略**：当资源紧张时，通过拒绝策略（如 `AbortPolicy`）快速失败，而不是让任务无限堆积。
4.  **监控队列大小**：实时监控 `workQueue.size()`，及时发现积压。
5.  **避免任务内存泄漏**：确保任务本身不会持有大对象或导致内存泄漏。

### 7. 线程池监控与调优实践？

**监控指标（可通过 JMX 或自定义监控）：**
*   `getPoolSize()`：当前线程数。
*   `getActiveCount()`：正在执行任务的线程数。
*   `getQueue().size()`：队列中的任务数。
*   `getCompletedTaskCount()`：已完成任务数。
*   `getLargestPoolSize()`：曾达到的最大线程数。

**调优实践：**
1.  **压力测试**：模拟生产流量，观察监控指标。
2.  **调整参数**：
    *   如果 `queue.size()` 经常很大 → 增加 `corePoolSize` 或 `maximumPoolSize`，或优化任务处理速度。
    *   如果 `activeCount` 接近 `maximumPoolSize` → 说明线程池在频繁扩容，可能需要增加 `corePoolSize`。
    *   如果频繁触发拒绝策略 → 需要增加线程数、队列大小，或优化业务逻辑。
3.  **动态调优**：高级场景下，可实现动态调整线程池参数（如 `setCorePoolSize()`）。

### 8. 线程池与 `ForkJoinPool` 区别？

| 特性 | `ThreadPoolExecutor` | `ForkJoinPool` |
| :--- | :--- | :--- |
| **设计目的** | 通用任务执行 | **分治算法**（Divide and Conquer），如递归、并行流（`parallelStream`）。 |
| **任务模型** | `Runnable` / `Callable` | `RecursiveTask`（有返回值） / `RecursiveAction`（无返回值）。 |
| **工作窃取** | ❌ 不支持 | ✅ **核心特性**！空闲线程会从其他线程的队列尾部“窃取”任务，提高 CPU 利用率。 |
| **适用场景** | I/O 任务、定时任务、通用异步处理。 | CPU 密集型、可分解的计算任务（如归并排序、矩阵运算、大数据处理）。 |
| **创建方式** | 显式构造或 `Executors` | `ForkJoinPool.commonPool()`（公共池）或自定义构造。 |
| **默认线程数** | `Executors` 方法有不同 | `commonPool` 默认线程数为 `N_cpu - 1`。 |

**简单说**：
*   **普通线程池**：适合处理**独立的、不可分割的任务**。
*   **`ForkJoinPool`**：适合处理**一个大任务可以拆分成多个小任务并行执行，最后合并结果**的场景，利用“工作窃取”机制高效利用多核 CPU。

---

### 总结

线程池是 Java 并发编程的基石。掌握其原理、参数调优、拒绝策略、关闭方式和监控方法，是构建高性能、高可用服务的必备技能。记住：**避免使用 `Executors` 的快捷方法，优先使用 `ThreadPoolExecutor` 自定义有界线程池**，这是生产环境的最佳实践。
