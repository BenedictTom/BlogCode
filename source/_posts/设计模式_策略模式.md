### 什么是策略模式（Strategy）

**思想**：将一组可互换的算法（行为）封装为独立的策略类，对外暴露统一的抽象接口；在运行时按需选择具体策略，而不修改使用策略的客户端代码。

- **角色**：
  - **Strategy（策略接口）**：约束策略的统一行为。
  - **ConcreteStrategy（具体策略）**：不同算法/行为的落地实现，可互换。
  - **Context（上下文）**：持有策略引用，对外屏蔽算法细节，只负责在合适的时机调用策略。
- **解决的问题**：
  - 消除 if-else/switch 大量分支判断（按类型执行不同逻辑）。
  - 在不改动客户代码的前提下，按配置/数据动态切换算法。
- **关键好处**：
  - 开闭原则（新增策略无需修改旧代码）。
  - 单一职责（每个策略只关心一种算法）。
  - 可测试性（策略可单测、可热插拔）。
- **代价**：
  - 类数量上升，需要合适的策略发现/注册机制。

---

### 适用场景

- **支付/计费场景**：微信/支付宝/银联等多种支付渠道；不同计费规则（包年、包月、按量）。
- **营销/优惠计算**：满减、折扣、立减、会员价等多种优惠策略。
- **排序/路由/压缩/序列化**：按不同算法切换实现。
- **风控/审核**：低风险、一般、严格等不同策略链路。
- **多租户/AB 实验**：按租户或实验桶动态切换行为。

### 经典实现（OOP 版）

以“订单优惠计算”为例：

```java
// 策略抽象
public interface DiscountStrategy {
    /** 返回折后价格 */
    double apply(double originPrice);
}

// 具体策略：无折扣
public class NoneDiscountStrategy implements DiscountStrategy {
    @Override
    public double apply(double originPrice) {
        return originPrice;
    }
}

// 具体策略：满减 100-20
public class FullReductionStrategy implements DiscountStrategy {
    @Override
    public double apply(double originPrice) {
        return originPrice >= 100 ? originPrice - 20 : originPrice;
    }
}

// 具体策略：9 折
public class PercentageStrategy implements DiscountStrategy {
    private final double rate; // 0~1
    public PercentageStrategy(double rate) { this.rate = rate; }
    @Override
    public double apply(double originPrice) { return originPrice * rate; }
}

// 上下文：对外只暴露计算方法
public class DiscountContext {
    private DiscountStrategy strategy;
    public DiscountContext(DiscountStrategy strategy) { this.strategy = strategy; }
    public void setStrategy(DiscountStrategy strategy) { this.strategy = strategy; }
    public double calculate(double originPrice) { return strategy.apply(originPrice); }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        DiscountContext ctx = new DiscountContext(new NoneDiscountStrategy());
        System.out.println(ctx.calculate(120)); // 120.0

        ctx.setStrategy(new FullReductionStrategy());
        System.out.println(ctx.calculate(120)); // 100.0

        ctx.setStrategy(new PercentageStrategy(0.9));
        System.out.println(ctx.calculate(120)); // 108.0
    }
}
```

要点：策略对外统一为 `apply`；`Context` 可在运行时切换策略，消除了硬编码分支。

### Spring 实战：基于策略注册表的自动装配

在实际项目中，常见做法是借助 Spring 容器将所有策略实现自动注册为 Bean，并通过一个 `Map<String, DiscountStrategy>` 按“策略编码”检索。

```java
// 1) 策略接口
public interface DiscountStrategy {
    String code();             // 策略编码，如 "NONE"、"FULL_REDUCTION"、"PERCENTAGE_90"
    double apply(double originPrice);
}

// 2) 策略实现
@Component
public class NoneDiscountStrategy implements DiscountStrategy {
    @Override public String code() { return "NONE"; }
    @Override public double apply(double originPrice) { return originPrice; }
}

@Component
public class FullReductionStrategy implements DiscountStrategy {
    @Override public String code() { return "FULL_REDUCTION"; }
    @Override public double apply(double originPrice) { return originPrice >= 100 ? originPrice - 20 : originPrice; }
}

@Component
public class Percentage90Strategy implements DiscountStrategy {
    @Override public String code() { return "PERCENTAGE_90"; }
    @Override public double apply(double originPrice) { return originPrice * 0.9; }
}

// 3) 策略工厂/注册表
@Component
public class DiscountStrategyRegistry {
    private final Map<String, DiscountStrategy> codeToStrategy;

    public DiscountStrategyRegistry(List<DiscountStrategy> strategies) {
        this.codeToStrategy = strategies.stream().collect(Collectors.toMap(DiscountStrategy::code, s -> s));
    }

    public DiscountStrategy getByCode(String code) {
        DiscountStrategy strategy = codeToStrategy.get(code);
        if (strategy == null) throw new IllegalArgumentException("Unknown strategy: " + code);
        return strategy;
    }
}

// 4) 业务 Service 使用
@Service
public class OrderService {
    private final DiscountStrategyRegistry registry;
    public OrderService(DiscountStrategyRegistry registry) { this.registry = registry; }

    public double calcPayable(double originPrice, String strategyCode) {
        return registry.getByCode(strategyCode).apply(originPrice);
    }
}
```

要点：
- 新增策略 = 新增一个 `@Component` 实现类；无需改动现有分发逻辑，符合开闭原则。
- 策略选择可以来自配置、订单字段、AB 实验分桶等。

### 函数式/无类实现（Java 8+）

对轻量逻辑，可直接用 `Function<Double, Double>` 表达策略，并放入 `Map`。

```java
public class LambdaStrategyDemo {
    private static final Map<String, Function<Double, Double>> STRATEGIES = new HashMap<>();
    static {
        STRATEGIES.put("NONE", p -> p);
        STRATEGIES.put("FULL_REDUCTION", p -> p >= 100 ? p - 20 : p);
        STRATEGIES.put("PERCENTAGE_90", p -> p * 0.9);
    }

    public static double calc(double originPrice, String code) {
        Function<Double, Double> fn = STRATEGIES.get(code);
        if (fn == null) throw new IllegalArgumentException("Unknown strategy: " + code);
        return fn.apply(originPrice);
    }
}
```

优点：更少样板代码；缺点：语义性、可测试性稍弱（没有类型化接口）。

### 与其他模式的关系

- **策略 vs 状态**：二者结构相似；策略强调“可替换的算法”，状态强调“对象内部状态迁移导致行为变化”。
- **策略 vs 工厂**：工厂负责“创建哪种策略”；策略负责“如何执行”。
- **策略 vs 责任链**：策略是“二选一/多选一”；责任链是“依序尝试/累加处理”。

---

### 实践经验与建议

- **命名**：策略名用业务语义（如 `Percentage90Strategy`）+ `code()` 唯一标识。
- **落地**：
  - 小型项目可直接 `Map<code, strategy>`；
  - 中大型项目用 Spring 自动收集 + 注册表；
  - 超多策略时可结合配置中心/插件化加载。
- **扩展**：策略可与校验器、组合策略（多个策略叠加）一起使用。

    public interface SearchExecutor<T> {
        Set<T> execute(String collection, List<String> questions,
                       List<BaseVector> dense, List<BaseVector> sparse,
                       double threshold, SseEmitterUTF8 emitter) throws Exception;
    }

    private <T> Set<T> searchWithStrategy(
            SseEmitterUTF8 emitter,
            List<String> questions,
            List<Integer> kbIds,
            RagStrategyEnum strategy,
            String collectionPrefix,
            Function<RagStrategyEnum, SearchExecutor<T>> executorFactory,
            double threshold
    ) {
        Set<T> result = new HashSet<>();
        sendSse(emitter, "searching");

        try {
            List<BaseVector> dense = null, sparse = null;
            if (strategy != RagStrategyEnum.KEYWORD) {
                Map<String, Object> vectors = getVector(questions);
                dense = (List<BaseVector>) vectors.get("dense");
                sparse = (List<BaseVector>) vectors.get("sparse");
            }

            SearchExecutor<T> executor = executorFactory.apply(strategy);

            for (Integer kbId : kbIds) {
                try {
                    result.addAll(executor.execute(
                            collectionPrefix + kbId, questions, dense, sparse, threshold, emitter
                    ));
                } catch (Exception ex) {
                    log.warn("{} 查询失败，跳过 kbId={}，原因：{}", strategy, kbId, ex.getMessage(), ex);
                }
            }
        } finally {
            sendSse(emitter, "searching_done");
        }
        return result;
    }

                },

### 这段代码如何体现策略模式

- **Strategy（策略接口）**：`SearchExecutor<T>` 定义了统一行为 `execute(...)`，不同检索算法（关键词/向量/混合）都以该接口为约束。
- **ConcreteStrategy（具体策略）**：在调用处通过 `executorFactory.apply(strategy)` 获得具体执行器；在 `getRelatedQaResult` 与 `getRelatedArticleResult` 中，分别用 `switch (s)` 返回不同的 Lambda，实现了三种策略：
  - `KEYWORD`：仅关键词检索（不需要向量）
  - `VECTOR`：仅向量检索
  - `HYBRID`：关键词 + 向量混合检索
  这些 Lambda 就是具体策略的实现，符合 `SearchExecutor<T>` 的签名。
- **Context（上下文）**：`searchWithStrategy(...)` 承担上下文角色。
  - 负责前后流程编排：SSE 事件通知（`sendSse`）、按需准备向量（仅非 `KEYWORD` 时才计算）、遍历 `kbIds` 聚合结果、异常隔离。
  - 不关心算法细节，将“怎么查”的责任交给策略执行器。
- **Strategy Selector（策略选择器）**：`RagStrategyEnum strategy` + `executorFactory: Function<RagStrategyEnum, SearchExecutor<T>>` 组合决定“用哪种策略”。
  - 策略选择与策略实现解耦，`searchWithStrategy` 仅依赖抽象 `SearchExecutor<T>`。

---

### 关键设计点与收益

- **消除分支地狱**：没有到处的 `if/else` 分支，策略分发集中在 `switch (s)` 构造具体执行器（也可替换为 Map 注册表）。
- **开闭原则**：新增检索算法时，只需新增一个策略（新增一个 Lambda 或实现类），以及在 `executorFactory` 中挂接，无需修改 `searchWithStrategy` 主流程。
- **按需准备依赖**：仅在 `VECTOR/HYBRID` 时计算 `dense/sparse`，将准备工作放在 Context，策略端专注检索逻辑，职责清晰、可测试。
- **通用编排复用**：SSE 事件发送、异常隔离、跨 `kbIds` 聚合在 Context 统一处理，避免在每个策略中重复样板代码。
- **类型与通用性**：`SearchExecutor<T>` 与 `searchWithStrategy` 的泛型设计，支持返回 `QaRerankResponse` 或 `FileRerankResponse` 两类结果，最大化复用。

---

### 扩展与落地建议

- 将 `switch (s)` 升级为“策略注册表”（如 `Map<RagStrategyEnum, SearchExecutor<T>>` 或基于 Spring 注入的 Bean Map），以便按枚举/编码直接查表获得执行器，减少分发样板。
- 若策略变复杂，建议将 Lambda 替换为独立类实现 `SearchExecutor<T>`，便于单测与复用；复杂策略也可组合（如先关键词后向量）。
- `searchWithStrategy` 可进一步抽出“准备上下文对象”（包含向量、阈值、SSE 等），减少参数个数，便于演进。
