Kubernetes（常简写为K8s）是一个开源平台，用于自动化部署、扩展和管理容器化的应用。简单来说，它就像是一个智能的调度员和管理员，能够帮助开发者和运维人员更高效地运行应用程序。

### 容器化应用

首先，你需要了解什么是容器。容器是一种轻量级、可移植的运行环境，它包含了代码及其所有依赖项，确保应用可以在任何环境下以相同的方式运行。Docker就是一个广泛使用的容器技术的例子。

### Kubernetes的作用

1. **自动化部署**：Kubernetes可以让你定义你的应用应该如何部署，包括你想要运行多少个实例，以及它们应该如何被更新等。一旦配置好，Kubernetes就能自动处理这些部署任务。

2. **扩展性**：随着用户需求的变化，你的应用可能需要处理更多的请求。Kubernetes可以根据实际需要自动增加或减少应用的运行实例数量，这一过程叫做水平扩展。

3. **自我修复**：如果某个应用实例失败了，Kubernetes会注意到这个问题，并启动一个新的实例来替换它。这有助于保持服务的高可用性。

4. **负载均衡**：当有多个相同的容器在运行时，Kubernetes可以自动分配流量到这些容器中，保证不会出现某些容器过载而其他容器闲置的情况。

5. **资源优化**：通过有效地安排和管理容器，Kubernetes可以帮助企业更有效地使用计算资源，降低运营成本。

总之，Kubernetes提供了一套强大的工具，使得部署和管理大规模的应用程序变得更为简单和可靠。无论是小型项目还是大型分布式系统，都可以从Kubernetes所提供的功能中受益。

## Kubernetes 特性

**服务发现和负载均衡**
Kubernetes 可以使用 DNS 名称或自己的 IP 地址公开容器，如果进入容器的流量很大， Kubernetes 可以负载均衡并分配网络流量，从而使部署稳定。

**存储编排**
Kubernetes 允许你自动挂载你选择的存储系统，例如本地存储、公共云提供商等。

**自动部署和回滚**
你可以使用 Kubernetes 描述已部署容器的所需状态，它可以以受控的速率将实际状态 更改为期望状态。例如，你可以自动化 Kubernetes 来为你的部署创建新容器， 删除现有容器并将它们的所有资源用于新容器。

**自动完成装箱计算**
Kubernetes 允许你指定每个容器所需 CPU 和内存（RAM）。 当容器指定了资源请求时，Kubernetes 可以做出更好的决策来管理容器的资源。

**自我修复**
Kubernetes 重新启动失败的容器、替换容器、杀死不响应用户定义的 运行状况检查的容器，并且在准备好服务之前不将其通告给客户端。

**密钥与配置管理**
Kubernetes 允许你存储和管理敏感信息，例如密码、OAuth 令牌和 ssh 密钥。 你可以在不重建容器镜像的情况下部署和更新密钥和应用程序配置，也无需在堆栈配置中暴露密钥。


## 组件架构

Kubernetes 是一个复杂的系统，它由多个组件构成，这些组件共同工作以提供一个强大且灵活的平台来运行容器化的应用程序。以下是对 Kubernetes 组件架构的详细说明，并用通俗的概念进行解释。

### 主要组件

| 组件名称       | 描述                                                                                   | 通俗概念解释                                    |
|------------|--------------------------------------------------------------------------------------|---------------------------------------------|
| **etcd**   | 分布式键值存储，用于保存集群的所有数据。它是集群的“大脑”，所有其他组件都通过查询 etcd 来获取配置信息或状态。          | 数据库，就像一个存放所有重要信息的安全仓库。               |
| **API Server** | 所有对外交互的入口，是唯一可以访问 etcd 的组件。它处理 REST 请求，调度实际操作，并更新 etcd 中的状态。 | 前台接待员，负责接收所有的请求，并将它们分发给相应的部门处理。 |
| **Controller Manager** | 负责维护集群的状态，例如故障检测、自动扩展和滚动更新等。它监听集群状态的变化并作出相应调整。 | 自动化管理员，确保一切按照计划运行，如发现错误则尝试修复。      |
| **Scheduler** | 根据资源可用性选择在哪个节点上运行新的 Pod（一组一个或多个容器）。                      | 调度员，决定在哪里执行新任务，确保不会超出资源限制。           |
| **Kubelet** | 在每个节点上运行的代理，确保容器都在 Pod 中运行。它与 API Server 进行通信，管理本地容器的生命周期。 | 工作现场的监督员，确保所有的任务都被正确执行。                |
| **Kube-proxy** | 网络代理，负责为服务提供网络连接规则，实现服务发现和服务负载均衡。                         | 内部网络交通警察，确保数据包能够正确地到达目的地。             |

### 其他关键概念

- **Pod**：Kubernetes 中最小的部署单元，通常包含一个或多个紧密相关的容器。
  - 类似于一艘装有货物（应用）的小船，可能单独航行，也可能和其他小船一起编队航行。
  
- **Service**：定义了一组 Pod 的策略类型，以及如何访问这些 Pod（例如负载均衡）。
  - 可以想象成是一个电话总机，负责将外部来电转接到正确的内部线路。

- **Namespace**：提供一种机制在同一个物理集群中隔离资源和对象。
  - 就像是大楼里的不同楼层，每个公司占据一层楼，有自己的空间和设施。




## 搭建流程

### **一、Kubernetes 必须搭建集群吗？**  
**不一定**，但集群才是 Kubernetes 的真正价值所在。  

- **单机模式（Minikube/Kind）**：  
  - 适合初学者学习，功能完整但性能有限（比如用 `minikube` 在本地模拟一个单节点集群）。  
  - **意义**：练习基础操作（部署Pod、Service等），不需要真实服务器。  

- **生产环境集群**：  
  - 至少需要 **2个Node（1个Master+1个Worker）** 才能体现 Kubernetes 的核心能力（高可用、弹性扩缩容等）。  
  - **意义**：真实的业务场景需要多节点协作，比如一个节点挂了，Pod 会自动迁移到其他节点。  

**结论**：  
- 学习时可以用单机版，但想真正理解 Kubernetes，最终还是要搭集群。  

---

### **二、搭建 Kubernetes 集群的步骤（初学者版）**  
这里以 **Ubuntu 22.04** 为例，用 `kubeadm` 工具搭建一个最简集群（1个Master+1个Worker）。  

#### **准备工作**  
1. **2台云服务器或虚拟机**（建议配置：2核CPU/4GB内存/20GB磁盘，系统：Ubuntu 22.04）。  
2. 确保服务器之间能互通（关闭防火墙或放行端口）。  

---

#### **步骤1：所有节点初始化**  
在 **Master 和 Worker** 上执行：  
```bash
# 1. 关闭交换分区（Kubernetes 要求）  
sudo swapoff -a  
sudo sed -i '/ swap / s/^/#/' /etc/fstab  # 永久禁用  

# 2. 安装 Docker（容器运行时）  
sudo apt update  
sudo apt install -y docker.io  
sudo systemctl enable docker  

# 3. 安装 kubeadm/kubelet/kubectl  
sudo apt install -y apt-transport-https ca-certificates curl  
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.28/deb/Release.key | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg  
echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.28/deb/ /' | sudo tee /etc/apt/sources.list.d/kubernetes.list  
sudo apt update  
sudo apt install -y kubelet kubeadm kubectl  
sudo systemctl enable kubelet  
```

---

#### **步骤2：Master 节点初始化**  
只在 **Master** 上执行：  
```bash
# 初始化集群（注意替换你的内网IP）  
sudo kubeadm init --pod-network-cidr=10.244.0.0/16 --apiserver-advertise-address=<Master节点的内网IP>  

# 按提示操作：  
mkdir -p $HOME/.kube  
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config  
sudo chown $(id -u):$(id -g) $HOME/.kube/config  

# 安装网络插件（必需）  
kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml  
```

---

#### **步骤3：Worker 节点加入集群**  
在 **Worker** 上执行 **Master 初始化后提示的 join 命令**（类似以下格式）：  
```bash
sudo kubeadm join <Master内网IP>:6443 --token <token> --discovery-token-ca-cert-hash <hash>  
```

---

#### **步骤4：验证集群**  
回到 **Master** 执行：  
```bash
kubectl get nodes  # 看到 Master 和 Worker 状态为 Ready 即成功  
kubectl get pods -A  # 检查所有系统Pod是否运行正常  
```

---

### **常见问题**  
1. **节点无法加入集群**：检查防火墙是否放行了 `6443` 端口。  
2. **Pod 状态异常**：通常是网络插件未正确安装，重装 `flannel`。  
3. **资源不足**：虚拟机至少给 2GB 内存，否则 kubelet 可能崩溃。  

---

### **简化方案（快速体验）**  
如果觉得麻烦，可以直接用以下工具一键搭建：  
- **Minikube**（单机学习）：`minikube start`  
- **Kind**（本地集群）：`kind create cluster`  
- **K3s**（轻量级K8s）：`curl -sfL https://get.k3s.io | sh -`  



这个参数 `--pod-network-cidr=10.244.0.0/16` 是用来指定 **Kubernetes Pod 网络的 IP 地址范围**的，下面用通俗易懂的方式解释：

---

### **1. 什么是 `10.244.0.0/16`？**
- 这是一个 **私有IP地址段**（类似你家WiFi的192.168.1.0/24），专门分配给集群内的 Pod 使用。
- **`10.244.0.0`** 是起始地址，**`/16`** 表示子网掩码（前16位固定，后16位可变）：
  - 可分配的IP范围：`10.244.0.1` ~ `10.244.255.254`（共约6.5万个地址）。
  - 每个Pod会从这个范围内自动分配一个唯一IP。

---

### **2. 为什么需要这个参数？**
Kubernetes 要求：
- 每个Pod要有独立IP（就像每部手机需要一个电话号码）。
- 不同节点的Pod之间要能直接通信（就像同一个公司内部分机号能互拨）。
- 这个地址段不能和你的服务器真实IP冲突（比如服务器用的是`192.168.1.0/24`，这里用`10.244.0.0/16`就安全）。

---

### **3. 为什么是 `10.244.0.0/16`？**
- **Flannel网络插件**的默认配置就是使用这个网段（其他插件如Calico、Weave可能有不同要求）。
- 如果换成其他插件，可能需要修改：
  - Calico常用：`192.168.0.0/16`
  - Weave常用：`10.32.0.0/12`

---

### **4. 不设置会怎样？**
- 如果省略这个参数，后续安装网络插件（如Flannel）时会失败，Pod之间无法通信。

---

### **实际例子**
假设你按教程执行：
```bash
sudo kubeadm init --pod-network-cidr=10.244.0.0/16 --apiserver-advertise-address=192.168.1.100
```
- 集群会创建一个虚拟网络，所有Pod的IP都会是 `10.244.x.x`。
- 当你运行 `kubectl get pods -o wide` 时，看到的 `IP` 列就是从这个范围分配的：

```
NAME       READY   STATUS    IP           NODE
nginx-pod  1/1     Running   10.244.1.3   worker-node
```

---

### **修改风险提示**
除非你明确知道自己在做什么，否则不要随意更改这个网段：
- 如果和公司内网冲突，会导致网络异常。
- 如果网段太小（比如`/24`），Pod数量多了会IP不足。



## 网络插件

Flannel 和 Calico 是 Kubernetes 最常用的两种 **网络插件（CNI, Container Network Interface）**，它们的核心区别在于 **网络模型、功能特性和适用场景**。以下是详细对比：

---

### **1. 核心区别总结**
| 特性                | Flannel                              | Calico                                  |
|---------------------|--------------------------------------|-----------------------------------------|
| **网络模型**         | 简单的 Overlay 网络（默认 VXLAN）     | 纯三层路由（BGP）或可选的 IP-in-IP 封装 |
| **性能**            | 较低（有封装开销）                   | 更高（直接路由，无额外封装）            |
| **功能**            | 基础网络连通性                       | 高级网络策略（NetworkPolicy）、安全控制  |
| **适用场景**        | 小型集群、快速部署                   | 生产环境、需要安全策略的大规模集群      |
| **安装复杂度**      | 简单（一行命令）                     | 较复杂（需配置 BGP 或 IP 池）           |

---

### **2. 是否需要同时安装？**
- **绝对不要同时安装**！一个集群只能选择一个 CNI 插件，否则会导致网络冲突。
- 如果已经安装了 Flannel，想换 Calico，必须先删除 Flannel：
  ```bash
  kubectl delete -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml
  ```

---

### **3. 如何选择？**
#### **选择 Flannel 如果：**
- 你只需要让 Pod 之间能通信，不需要高级网络策略。
- 集群规模小，且对性能要求不高。
- 希望快速搭建一个测试环境。

#### **选择 Calico 如果：**
- 你需要 **网络策略（NetworkPolicy）** 实现 Pod 之间的访问控制（如隔离某些服务）。
- 集群规模大，且追求高性能（如金融、生产环境）。
- 需要支持 **BGP 路由**（与物理网络集成）。

---

### **4. 技术细节对比**
#### **Flannel 的工作原理**
1. 为每个节点分配一个子网（如 `10.244.1.0/24`）。
2. 使用 **VXLAN** 或 **host-gw** 封装流量，跨节点通信。
3. 没有内置的安全策略，依赖 Kubernetes 的 NetworkPolicy（功能有限）。

#### **Calico 的工作原理**
1. 为每个 Pod 分配 IP，通过 **BGP 协议** 在节点间广播路由（无需封装）。
2. 支持精细的 **NetworkPolicy**（如限制 Pod A 只能访问 Pod B）。
3. 可集成企业级防火墙（如 AWS Security Groups）。

---

### **5. 性能对比**
| 场景                | Flannel (VXLAN)       | Calico (BGP)          |
|---------------------|-----------------------|-----------------------|
| 跨节点延迟          | 较高（封装开销）      | 更低（直接路由）      |
| 吞吐量              | 中等                  | 更高                  |
| CPU 占用            | 较高（加密/解密）     | 更低                  |

---

### **6. 安装示例**
#### **安装 Flannel（最简单）**
```bash
kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml
```

#### **安装 Calico（功能更强）**
```bash
curl https://docs.projectcalico.org/v3.20/manifests/calico.yaml -O
kubectl apply -f calico.yaml
```

---

### **7. 如何验证？**
- 检查 Pod 网络是否正常：
  ```bash
  kubectl get pods -n kube-system  # 查看 flannel/calico 的 Pod 是否 Running
  kubectl get nodes -o wide       # 检查节点内网 IP 和 Pod CIDR
  ```

- 测试跨节点通信：
  ```bash
  kubectl run test --image=nginx --restart=Never
  kubectl exec -it test -- curl <另一个Pod的IP>
  ```

---

### **8. 常见问题**
#### **Q: 安装 Flannel 后，NetworkPolicy 不生效？**
- Flannel 默认不支持 NetworkPolicy，需额外安装 `flannel + canal`（Calico 的策略组件）。

#### **Q: Calico 报错 `BGP not established`？**
- 如果节点间无法通过 BGP 通信，改用 IP-in-IP 模式：
  ```yaml
  # 修改 calico.yaml 中的配置
  - name: CALICO_IPV4POOL_IPIP
    value: "Always"
  ```

---

### **总结**
- **Flannel**：适合快速搭建、简单场景，“能用就行”。
- **Calico**：适合生产环境，需要高性能和网络安全策略。
- **二选一**，不要混用！根据需求选择即可。


## NameSpace
NameSpace（命名空间）：用来对集群资源进行划分；默认只隔离资源，不隔离网络。 （-n）

```bash
# 查看命名空间
kubectl get ns

# 查看 部署了哪些应用
kubectl get pods -A
# 查看 默认命名空间（default）部署的应用
kubectl get pods

# 查看 某个命名空间（kubernetes-dashboard）部署的应用
kubectl get pod -n kubernetes-dashboard

# 创建命名空间
kubectl create ns hello
# 删除命名空间
kubectl delete ns hello
```

```yaml
# 版本号
apiVersion: v1
# 类型
kind: Namespace
# 元数据
metadata:
  # Namespace（命名空间） 的 名称
  name: hello-yaml
```

```bash
vi hello-yaml.yaml

# 将上述内容 粘贴进 hello-yaml.yaml

kubectl apply -f hello-yaml.yaml

kubectl get ns
# kubectl delete ns hello-yaml
# 配置文件创建的资源 用配置文件 删
kubectl delete -f hello-yaml.yaml
```



## Pod

Pod 是 Kubernetes 中最小的部署单元

---

### ✅ Kubernetes Pod 常用命令一览表（按使用频率排序）

| 排名 | 命令 | 用途说明 | 通俗解释 | 常见参数/示例 |
|------|------|----------|----------|-------------|
| 1 | `kubectl get pods` | 查看当前命名空间中所有 Pod 的状态 | 查看所有正在运行（或异常）的小船（Pod） | <ul><li>`-n <namespace>`：指定命名空间，如 `-n default`</li><li>`-A` 或 `--all-namespaces`：查看所有命名空间</li><li>`-o wide`：显示更多列（如节点IP）</li><li>`kubectl get pods -l app=nginx`：按标签筛选</li></ul> |
| 2 | `kubectl describe pod <pod-name>` | 查看某个 Pod 的详细信息（事件、容器状态、挂载、IP等） | 查看某艘船的“体检报告”和“日志记录” | <ul><li>常用于排查启动失败、崩溃等问题</li><li>可看到容器是否就绪、镜像拉取失败、资源不足等信息</li></ul> |
| 3 | `kubectl logs <pod-name>` | 查看 Pod 中容器的日志 | 听船员（容器）的“工作汇报”或“抱怨” | <ul><li>`-c <container-name>`：多容器 Pod 中指定某个容器</li><li>`-f` 或 `--follow`：实时跟踪日志输出</li><li>`--tail=100`：只看最后100行</li><li>`-n <namespace>`：指定命名空间</li></ul> |
| 4 | `kubectl delete pod <pod-name>` | 删除某个 Pod（如果是 Deployment 管理的，会自动重建） | 把一艘坏船击沉，系统会自动造一艘新的 | <ul><li>`--force --grace-period=0`：强制立即删除（慎用）</li><li>常用于触发滚动更新或解决卡住状态</li></ul> |

## 🌐 Kubernetes 工作负载（Workloads）详解

Kubernetes 的“工作负载”是指运行在集群中的一组 Pod（容器组），它们代表了你实际要运行的应用程序。Kubernetes 提供了多种**工作负载控制器（Workload Controllers）**，用来管理 Pod 的创建、更新、扩展和生命周期。

### 📊 一、Kubernetes 工作负载类型概览（树状图）

```
Kubernetes 工作负载（Workloads）
│
├── 🟢 Deployment        → 最常用！用于无状态应用（如 Web 服务）
│   └── 管理 ReplicaSet → 控制多个相同 Pod 副本
│
├── 🔵 StatefulSet       → 用于有状态应用（如数据库：MySQL, Redis）
│   └── 每个 Pod 有唯一身份（网络标识 + 存储）
│
├── 🟡 DaemonSet        → 每个节点运行一个 Pod（如日志采集、监控 agent）
│   └── 节点新增时自动部署
│
├── 🔴 Job              → 执行一次性的任务（如数据处理、备份）
│   └── 成功完成后退出
│
├── 🟣 CronJob          → 定时执行的 Job（类似 Linux 的 cron）
│   └── 按时间调度，如每天凌晨备份
│
└── ⚪ ReplicaSet        → Deployment 的底层实现（一般不直接使用）
    └── 确保指定数量的 Pod 副本运行
```

### 🧩 二、每种工作负载详细说明（附通俗比喻）

| 工作负载 | 用途 | 通俗比喻 | 适用场景 | 是否推荐直接使用 |
|--------|------|---------|----------|----------------|
| **Deployment** | 管理无状态应用，支持滚动更新、回滚、扩缩容 | 🏢 开发公司大楼：每层楼的办公室都一样，员工可以随便调换 | Web 服务（Nginx、Spring Boot）、API 服务 | ✅ 强烈推荐 |
| **StatefulSet** | 管理有状态应用，每个 Pod 有固定名称、网络标识和持久化存储 | 🏠 独栋别墅小区：每栋房子有固定门牌号，住的人不能乱换 | 数据库（MySQL、PostgreSQL）、分布式存储（etcd、ZooKeeper） | ✅ 有状态场景必用 |
| **DaemonSet** | 确保每个节点运行一个 Pod 副本 | 🚔 每个警察局都配一名交警：负责交通监控 | 日志收集（Fluentd）、监控代理（Node Exporter）、网络插件（Calico） | ✅ 特定场景使用 |

| **CronJob** | 按时间周期运行 Job | 📅 定闹钟的临时工：每天早上6点来扫地 | 定时备份、日志清理、周期性计算 | ✅ 定时任务首选 |
| **ReplicaSet** | 确保指定数量的 Pod 运行（Deployment 的底层） | 🔧 自动补人系统：少一个 Pod 就自动补上 | 一般由 Deployment 自动管理 | ❌ 不推荐直接使用 |

### 📄 完整配置文件（带注释）

```yaml
# 指定当前资源使用的 Kubernetes API 版本
# apps/v1 是 Deployment、StatefulSet 等工作负载控制器所在的 API 组
apiVersion: apps/v1

# 定义资源类型：这里是 Deployment（部署）
# Deployment 是最常用的工作负载控制器，用于管理无状态应用（如 Spring Boot）
kind: Deployment

# 元数据：描述这个 Deployment 本身的“身份信息”
metadata:
  # 名称：这是该 Deployment 的唯一标识符，在同一个命名空间内必须唯一
  name: backend

# 实际的配置内容（spec = specification）
spec:
  # 指定要运行多少个相同的 Pod 副本
  # 这里设置为 3，表示 Kubernetes 会确保始终有 3 个 backend Pod 在运行
  replicas: 3

  # 标签选择器：告诉 Deployment 如何找到它管理的 Pod
  # 它会查找所有带有标签 app: backend 的 Pod
  selector:
    matchLabels:
      app: backend

  # Pod 模板（template）：定义 Pod 的“蓝图”
  # 所有由这个 Deployment 创建的 Pod 都会基于这个模板生成
  template:
    # Pod 的元数据
    metadata:
      # 给每个 Pod 打上标签 app: backend
      # 这个标签必须和上面 selector.matchLabels 一致
      # 否则 Deployment 找不到它创建的 Pod！
      labels:
        app: backend

    # Pod 的实际规格（容器、存储、网络等）
    spec:
      # 容器列表（一个 Pod 可以包含多个容器，但这里只有一个）
      containers:
        # 第一个容器的定义
        - name: backend           # 容器名称（在 Pod 内唯一）
          image: my-springboot-api:v1.2  # 要运行的镜像（格式：镜像名:标签）
          # 提示：v1.2 是镜像版本号，建议使用具体版本而非 latest
```

### 🔍 核心概念通俗解释

| 概念 | 通俗比喻 | 说明 |
|------|----------|------|
| **Deployment** | 🏭 自动化工厂 | 负责创建、更新、扩缩、回滚一组相同的 Pod，确保它们按预期运行 |
| **replicas: 3** | 🏗️ 工厂生产 3 艘一模一样的船 | 确保始终有 3 个 Pod 实例在运行，如果某个挂了，自动重建 |
| **selector.matchLabels** | 🔍 工厂的“识别标签” | 工厂只管理贴有 `app: backend` 标签的 Pod，不会误管其他船 |
| **template.metadata.labels** | 🏷️ 给每艘船贴上标签 | 新造的船必须贴上 `app: backend`，否则工厂找不到它 |
| **containers.image** | ⚙️ 船上的发动机型号 | 指定运行哪个应用镜像，这里是 Spring Boot 后端服务 v1.2 版本 |

### ✅ 这个配置会做什么？

当你运行：

```bash
kubectl apply -f backend-deployment.yaml
```

Kubernetes 会：

1. 创建一个名为 `backend` 的 Deployment。
2. 根据模板创建 **3 个 Pod**，每个 Pod 都运行 `my-springboot-api:v1.2` 镜像。
3. 给每个 Pod 打上标签 `app: backend`。
4. 持续监控这 3 个 Pod：
   - 如果某个 Pod 崩溃 → 自动创建一个新的。
   - 如果你修改镜像为 `v1.3` → 触发**滚动更新**，逐步替换旧 Pod。
   - 如果更新出问题 → 可以用 `kubectl rollout undo` 回滚到 `v1.2`。

---

### ⚠️ 注意事项

1. **标签必须匹配！**
   - `selector.matchLabels.app` 必须和 `template.metadata.labels.app` 一致。
   - 否则 Deployment 无法管理 Pod，会报错。

2. **不要使用 `latest` 镜像标签**
   - `image: my-springboot-api:latest` 会导致更新不可控。
   - 推荐使用语义化版本：`v1.2`, `v1.3` 等。

3. **生产环境建议添加更多配置**
   - 资源限制（CPU/内存）
   - 健康检查（liveness/readiness probes）
   - 环境变量、密钥、配置文件挂载等

### ✅ 下一步建议：配合 Service 使用

单独的 Deployment 只能创建 Pod，但无法对外提供访问。你需要一个 `Service` 来暴露服务：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  selector:
    app: backend        # 找到所有 app=backend 的 Pod
  ports:
    - protocol: TCP
      port: 80          # Service 对外端口
      targetPort: 8080  # Pod 中 Spring Boot 的实际端口
  type: ClusterIP       # 或 NodePort / LoadBalancer
```

然后其他服务（如前端）就可以通过 `http://backend-service:80` 访问你的 Spring Boot 后端。

> **声明式 API + 控制器模式 + 标签选择器 = 自动化、高可用的应用管理**

## 🌐 Kubernetes Service 是什么？（一句话理解）

> **Service 是 Pod 的“稳定门面”或“智能路由器”**，它提供一个固定的 IP 和 DNS 名称，将请求自动转发给后端的一组 Pod，并实现负载均衡。

即使 Pod 重启、IP 变化、扩缩容，Service 的访问地址始终不变 ✅

## 📊 Service 常用命令表（按使用频率排序）

| 排名 | 命令 | 用途说明 | 通俗解释 | 示例 |
|------|------|----------|----------|------|
| 1 | `kubectl get services` 或 `kubectl get svc` | 查看当前命名空间中所有 Service | 查看所有“服务入口” | `kubectl get svc` |
| 2 | `kubectl describe svc <service-name>` | 查看某个 Service 的详细信息（后端 Pod、端口、选择器等） | 查看“门卫”的工作日志和管理的员工名单 | `kubectl describe svc backend-service` |
| 3 | `kubectl create service <type>` | 手动创建一个 Service（不常用，推荐用 YAML） | 手工造一个“接待台” | `kubectl create service clusterip my-svc --tcp=80:8080` |
| 4 | `kubectl apply -f service.yaml` | 根据 YAML 文件创建或更新 Service | 按图纸建造或升级“接待台” | `kubectl apply -f backend-service.yaml` |
| 5 | `kubectl delete svc <service-name>` | 删除某个 Service | 拆除一个“接待台” | `kubectl delete svc backend-service` |
| 6 | `kubectl get endpoints <service-name>` | 查看 Service 实际绑定的 Pod IP 列表 | 查看“门卫”当前正在转发给哪些“员工” | `kubectl get endpoints backend-service` |
| 7 | `kubectl edit svc <service-name>` | 在线编辑 Service 配置（生产慎用） | 直接修改“接待台”的工作规则 | `kubectl edit svc backend-service` |
| 8 | `kubectl expose deployment <deploy-name> --port=80 --target-port=8080 --name=<svc-name>` | 为已有 Deployment 快速创建 Service | 给一个已有的“工厂”加一个“接待台” | `kubectl expose deployment backend --port=80 --target-port=8080 --name=backend-svc` |

### 🔍 命令详细说明

#### 1. `kubectl get svc`
- 查看所有 Service 的基本信息：名称、类型、集群 IP、外部 IP、端口、年龄。
- 常用参数：
  - `-n <namespace>`：指定命名空间
  - `-A` 或 `--all-namespaces`：查看所有命名空间的 Service
  - `-o wide`：显示更多列（如选择器）

```bash
kubectl get svc -n default
kubectl get svc -A
```

#### 2. `kubectl describe svc <name>`
- 查看 Service 的详细信息，包括：
  - `Selector`：匹配哪些 Pod（如 `app=backend`）
  - `Endpoints`：当前实际绑定的 Pod IP 列表
  - `Port` 映射关系
  - 事件（Events）

```bash
kubectl describe svc backend-service
```

#### 3. `kubectl apply -f service.yaml`
- 推荐方式，通过声明式 YAML 创建或更新 Service。

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  selector:
    app: backend
  ports:
    - protocol: TCP
      port: 80
      targetPort: 8080
  type: ClusterIP  # 或 NodePort, LoadBalancer
```

```bash
kubectl apply -f service.yaml
```

#### 4. `kubectl expose ...`
- 快速为已有 Deployment 创建 Service。

```bash
kubectl expose deployment backend \
  --port=80 \
  --target-port=8080 \
  --name=backend-svc \
  --type=NodePort
```

#### 5. `kubectl get endpoints`
- 查看 Service 实际转发到哪些 Pod 的 IP 和端口。
- 如果 endpoints 为空，说明没有 Pod 匹配 `selector`，常见于标签不一致。

```bash
kubectl get endpoints backend-service
# 输出示例：10.244.1.5:8080, 10.244.2.6:8080
```

---

### 🧩 Service 类型与访问方式

| 类型 | 用途 | 外部能否访问 |
|------|------|-------------|
| `ClusterIP`（默认） | 集群内部访问 | ❌ 仅集群内 |
| `NodePort` | 通过节点 IP + 固定端口暴露 | ✅ 可以，端口范围 30000-32767 |
| `LoadBalancer` | 云厂商提供公网负载均衡器 | ✅ 公网可访问（需云支持） |
| `Ingress` | 更高级的 HTTP 路由（基于域名/路径） | ✅ 推荐用于 Web 服务 |

---

### ✅ 实际使用建议

1. **开发环境**：用 `NodePort` 快速测试
2. **生产环境**：用 `Ingress` + `ClusterIP` 实现 HTTPS、域名路由
3. **微服务间调用**：用 `ClusterIP`，通过 DNS 名称访问（如 `http://backend-service:80`）
